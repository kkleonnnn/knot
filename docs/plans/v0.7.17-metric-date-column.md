# v0.7.17 — 语义层第十八刀：metric 显式 date_column（修 `sta_date` 时间窗编译 gap）（C1~C3）

> **协议**：Loop Protocol v3 完整三阶段（改 metric 模型 + 编译器 date-col 解析，守护者逐 commit 复核）
> **触发**：v0.7.16 后「价值自测验证」（资深拍 A）—— grounded 真实 OHX catalog 发现编译器 date-col 正则
> `^(date|dt|ds|day|日期|stat_date|biz_date)$` **匹配不到 OHX 全表实际日期列 `sta_date`**（正则有 `stat_date` 带 t，
> 实际列 `sta_date` 无 t）→ `_resolve_date_col` 返 None → 时间窗 metric `raise CompileError` → 回退 LLM。
> 几乎所有 BI 问题带时间窗 → **语义层即使开启 + 注册 metric，时间窗查询命中 ≈ 0%**。
> **资深 AskUserQuestion 拍「两者都做」**：显式 `date_column` 字段（优先，契合注册表「显式定义」哲学）+ 加宽正则（fallback）。

---

## 1. 背景与定调

价值自测（v0.7.16 后资深拍 A）**第一发现即 blocker**：编译器 date-col 启发式正则与真实 catalog 列名不匹配。
- `_resolve_date_col(["sta_date"])` → `None`（grounded 实证：[compile_helpers.py:21](knot/services/semantic/compile_helpers.py)）
- OHX 全表（daily/weekly/monthly/future_market/spot_market/dwd_user_reg）日期列均为 `sta_date` → 正则命中 0
- 后果：时间窗 metric 编译 raise → 回退 LLM → 语义层时间窗价值不释放

**根因**：date 列靠**正则猜**是脆弱启发式（任何 catalog 列名变体都会再 miss：`sta_date`/`ymd`/`period`/`日`…）。
注册表的哲学是**显式定义口径** —— date 列同理应可显式声明，不靠猜。

**修法（资深拍「两者都做」）**：
1. **显式 `date_column` metric 字段**（优先）—— admin 注册时声明日期列（如 `sta_date`），编译时直接用，0 猜测
2. **加宽正则 `_date$` 后缀**（fallback）—— 未声明 date_column 时 regex 兜底（`sta_date`/`stat_date`/`biz_date` 全 `_date$` 命中）

## 2. 设计

### 2.1 metric 模型 +`date_column` 字段

- schema.sql metrics 表 +`date_column TEXT DEFAULT ''`（在 `dimensions` 后；非密 — 列名非机密，R-SL-5 sustained）
- base.py 幂等 ALTER（镜像 v0.7.6 `restored_from_audit_id` / v0.6.2.5 `active_catalog_id` 模式 —— `ADD COLUMN date_column TEXT DEFAULT ''`，存量 metric 自动得空串=未声明=regex fallback）
- metric_repo `_COLS`（SELECT 列）+`_UPDATABLE`（create/update 白名单）+ date_column
- **OOS-1 死线 sustained**：date_column 是列名（catalog 水平切分内），**非** tenant_id；`_FORBIDDEN_KEYS` 不变

### 2.2 compiler date-col 解析（显式优先 + regex fallback）

新 helper（compile_helpers）：
```python
def _resolve_metric_date_col(m: dict, fallback_dims=None) -> str | None:
    """显式 date_column 优先（v0.7.17）；未声明 → regex fallback on dims（保 byte-equal）。"""
    explicit = (m.get("date_column") or "").strip()
    if explicit:
        return explicit
    dims = fallback_dims if fallback_dims is not None else _json_list(m.get("dimensions"))
    return _resolve_date_col(dims)
```

**4 个 date-col 解析点全接入**（metric `m` 在各点均可达）：
| 点 | 现状 | 改为 |
|---|---|---|
| compiler 单对象 [L208](knot/services/semantic/compiler.py) | `_resolve_date_col(sorted(allowed_dims))` | `_resolve_metric_date_col(metrics_by_name[lf.metrics[0]], sorted(allowed_dims))` |
| compiler 跨对象 [L258](knot/services/semantic/compiler.py) | `_resolve_date_col(sorted(obj_dims.get(base, set())))` | `_resolve_metric_date_col(metrics_by_name[lf.metrics[0]], sorted(obj_dims.get(base, set())))` |
| compile_helpers `_scalar_subquery` | `_resolve_date_col(_json_list(m.get("dimensions")))` | `_resolve_metric_date_col(m)` |
| multi_base `_build_dimensional_sql` | `_resolve_date_col(sorted(avail))` | `_resolve_metric_date_col(m, sorted(avail))` |

> 单/跨对象单 base 聚合：所有 metric 共享 base，date_column 应一致（base 表的列）→ 取 `lf.metrics[0]` 的；
> `fallback_dims` 保留原 union-dims regex 兜底 → **未声明 date_column 时 byte-equal**（regex 行为不变 + 加宽仅 ADD 匹配）。

### 2.3 fallback regex 加宽 = **两遍解析（守护者 Stage 3 承重修订 + 执行者机制精化）**

⚠️ **草案原拟单 regex `^(date|dt|ds|day|日期)$|_date$` 是错的**（两处实证）：
1. **`re.match` 锚定串首 → `_date$` 不做后缀匹配**（`re.match("_date$","sta_date")` → None，因 "sta_date" 首字符非 `_`）⟹ 单 regex 加宽**根本不命中 `sta_date`**（NO-OP，gap 没修）—— 执行者实证精化。
2. 即便用 `.*_date$`，**单遍首个匹配会让 `_date$` 列抢占旧 exact 列**（守护者 finding）：`["amount_date","stat_date"]` 旧返 `stat_date`（exact），单遍 `_date$` 优先则漂 `amount_date` = **选错日期列（静默错数据，非 cosmetic）**。

**修法 = 两遍解析（strictly additive）**：
```python
_DATE_COL_EXACT_RE = re.compile(r"^(date|dt|ds|day|日期|stat_date|biz_date)$", re.I)  # ← 当前 regex 一字不改
def _resolve_date_col(allowed_dims):
    for d in allowed_dims:                                   # pass1: 旧 exact（order+result 逐字保留）
        if _DATE_COL_EXACT_RE.match(str(d).strip()):
            return str(d).strip()
    for d in allowed_dims:                                   # pass2: _date$ 后缀（仅 pass1 全 None 时填）
        if str(d).strip().lower().endswith("_date"):
            return str(d).strip()
    return None
```
- ⚠️ **pass1 必须保留 `stat_date|biz_date`**（草案拟删之靠后缀覆盖 → `["amount_date","stat_date"]` 两遍下漂 `amount_date` 实证）。pass1 = 当前 regex **逐字不改**。
- **pass2 用 `.endswith("_date")` 字符串法**（非 `_date$` regex —— re.match 锚首失效）；小写比较 + 原样返。
- ⟹ **旧已 exact 命中 → pass1 逐字同列（byte-equal 铁保）**；仅旧返 None 的（`sta_date`）被 pass2 填。**漂移归零** —— 这正是 Stage 2 R-SL-142「exact ==」在多日期列场景成立的前提。
- **不过匹配**：`city`/`region`/`candidate`/`city_candidate`/`update_state`（无 `_date` 后缀）pass1+pass2 全不命中 — 安全（守护者盲刺保留）。

## 6. 验收标准

1. **显式优先**：metric `date_column="sta_date"` + LogicForm time → SQL 含 `o.sta_date BETWEEN '...' AND '...'`（即使维度无 regex-匹配列）
2. **regex fallback（两遍 pass2）**：metric date_column 空 + 维度含 `sta_date` → `.endswith("_date")` 命中 → 注入（修 OHX gap）
3. **byte-equal（未声明 + regex-匹配维度）**：date_column 空 + 维度 `["date","city"]` → 编译 SQL 与 v0.7.16 **exact `==`**（0 漂移）；**多日期列 `["created_date","dt"]` → 仍返 `"dt"`（order-drift 守护，R-SL-142）**
4. **不过匹配**：维度 `["city"]`（或 `["city_candidate"]`）+ date_column 空 + time → 仍 raise（无 `_date` 后缀）
5. metrics 列集 15（+date_column）；`test_metrics_schema_no_tenant` 同步；OOS-1 sustained（0 tenant_id）
6. 5 源点 0.7.16→0.7.17 + CHANGELOG count==1 + check_file_sizes + Contract 9 + CI 8 绿
