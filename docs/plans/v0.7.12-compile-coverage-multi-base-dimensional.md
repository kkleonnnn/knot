## 1. 决议（承重刀 — JOIN 类型已由资深拍定）

**一句话**：让**多 base + 维度**（如「各城市 GMV 和 DAU」，gmv@orders + dau@users 按 city 切）走确定性编译 —— 用**维度并集驱动 + LEFT JOIN 各聚合**（Option U），承接 v0.7.11 标量多 base。

### ⭐ 核心架构（grounded main `b60bc3c`）

**JOIN 类型抉择（资深拍 Option U）—— grounded 三选一**：
- **现有系统已有「聚合后 LEFT JOIN 子查询」模式**（`llm_prompt_builder.py` L51-54 反 fan-out prompt：`LEFT JOIN (SELECT uid, SUM(amt) FROM deposits GROUP BY uid) d`）→ LEFT JOIN 聚合子查询是 Doris-proven 生产模式。
- **FULL OUTER JOIN 现有代码 0 用**（未测构造，Doris 支持仅 kk 知）→ 弃。
- **Option U = 维度并集驱动**：各 base 独立聚合 + **维度并集（UNION）做驱动** + LEFT JOIN 各聚合 → **正确（对称不丢）+ Doris-proven（仅 UNION+LEFT，0 FULL OUTER）**。

```sql
-- 「各城市 GMV 和 DAU」（gmv@orders + dau@users，按 city）
SELECT dim.city, t0.gmv, t1.dau FROM (
  SELECT DISTINCT o.city FROM shop.orders o WHERE o.status='paid'   -- 维度域：base0
  UNION
  SELECT DISTINCT o.city FROM shop.users o                          -- 维度域：base1
) dim
LEFT JOIN (SELECT o.city, SUM(o.pay_amount) AS gmv FROM shop.orders o WHERE o.status='paid' GROUP BY o.city) t0
  ON dim.city = t0.city
LEFT JOIN (SELECT o.city, COUNT(DISTINCT o.user_id) AS dau FROM shop.users o GROUP BY o.city) t1
  ON dim.city = t1.city
LIMIT 1000
```

**⭐ 基数 BY CONSTRUCTION（R-SL-108）**：维度并集 driver 1 行/维度组合 + 各聚合子查询 1 行/维度组合 → LEFT JOIN on 维度 = **1:1 不膨胀**（聚合-**后**-JOIN，非 v0.7.2「JOIN-后-聚合」膨胀坑）。

**grounded 双安全门**（决定形态可行性）：
- `is_cartesian`：C2 只 flag `kind=="CROSS"`；LEFT JOIN 有 ON → 不 flag；UNION 分支单表无 join → 不触发。**过** ✓
- `_is_safe_sql`：outer Select 根 + `Union` + 嵌套 `Subquery` 全 ∈ allowed_roots（grounded doris.py:328）+ 无写节点。**过** ✓

### In-Scope
- compiler：`_build_multi_base_sql` 分 scalar（v0.7.11，无维度）vs dimensional（v0.7.12，维度并集驱动）
- parser：`_validate_hit` 放宽（多 base + 维度，compiler 共同维度 authoritative）+ prompt 教共同维度

### OOS（推后续刀 / 已知限制）
- **非共同维度**（某维度只在部分 base）→ raise 回退（无法聚合后 JOIN）
- 多 base + having/window/qualify/filters → raise（单 base grain 概念 / 归属歧义）
- **NULL 维度值**：用 `=`（非 null-safe `<=>`）—— 业务维度（city/date/region）多非 NULL；NULL-dim 边界已知限制（守护者可议是否升 `<=>`）

## 2. 设计

### 2.1 compiler 分发（scalar vs dimensional）

`_build_multi_base_sql` 重构为分发器（guards 二者共用 + 按 dims 分支）：
```python
def _build_multi_base_sql(lf, metrics_by_name, tables, time_ctx):
    # guards（scalar + dimensional 共用）—— 全矩阵 raise 守 having/window/qualify/filters（R-SL-101 sustained）
    if lf.having or lf.window or lf.qualify or lf.filters:
        raise CompileError("多 base + having/window/qualify/filters → 回退（单 base grain 概念）")
    if lf.time and lf.time not in _TIME_KEYS:
        raise CompileError(f"未知 time 枚举 {lf.time!r}")
    if not lf.dimensions:
        return _build_multi_base_scalar_sql(lf, metrics_by_name, tables, time_ctx)   # v0.7.11 标量（byte-equal）
    return _build_multi_base_dimensional_sql(lf, metrics_by_name, tables, time_ctx)  # v0.7.12 维度并集驱动
```
v0.7.11 标量 body 平移 `_build_multi_base_scalar_sql`（**SQL 输出 byte-equal**，仅移函数）。

### 2.2 `_build_multi_base_dimensional_sql`（维度并集驱动 Option U · ⭐ per-metric 守护者 Stage 3 纠）

⭐ **守护者 Stage 3 承重纠正：per-metric agg（非 per-base 合并）**。Stage 1 草案曾按 base 分组、同 base 多 metric 合一 agg 用 `names[0]` 的 filters —— **bug**：同 base 两 metric filters 不同（如 `paid_gmv` status='paid' + `all_gmv` 无 filter）→ 合一 agg 用 `names[0]`(paid) 的 WHERE 算两者 → `all_gmv` 被错误过滤 = 静默算错。scalar 路径（v0.7.11）是 **per-metric**（各 metric 自己 filter）正确 → dimensional 也走 **per-metric**（每 metric 独立 agg + 自己的 filter），与 scalar 一致。

```python
def _build_multi_base_dimensional_sql(lf, metrics_by_name, tables, time_ctx):
    union_branches, agg_joins, sel_metrics = [], [], []
    for i, name in enumerate(lf.metrics):                 # ⭐ per-metric（与 scalar 一致；保序 t0/t1/…）
        m = metrics_by_name[name]
        avail = set(_json_list(m.get("dimensions")))
        for d in lf.dimensions:                           # 共同维度：每 dim ∈ 该 metric 可用维度（R-SL-107）
            if d not in avail:
                raise CompileError(f"维度 {d!r} ∉ metric {name!r} 可用维度（多 base 需共同维度）→ 回退")
        physical = _resolve_physical(m["base_object"], tables)   # HTTP/未匹配 → raise
        where = [str(f) for f in _json_list(m.get("filters"))]   # ⭐ 各 metric 自己的 filters（非 names[0] 共用）
        if lf.time:
            date_col = _resolve_date_col(sorted(avail))
            if date_col is None:
                raise CompileError(f"metric {name!r} base 无日期列但 lf.time 设定 → 回退")
            start, end = getattr(time_ctx, lf.time)
            where.append(f"o.{date_col} BETWEEN '{start}' AND '{end}'")
        dim_cols = ", ".join(f"o.{d}" for d in lf.dimensions)
        w = (" WHERE " + " AND ".join(where)) if where else ""
        union_branches.append(f"SELECT DISTINCT {dim_cols} FROM {physical} o{w}")
        agg = f"SELECT {dim_cols}, {m['caliber']} AS {name} FROM {physical} o{w} GROUP BY {dim_cols}"
        a = f"t{i}"
        # NULL-dim 已知限制：`=`（非 null-safe `<=>`）—— 业务维度多非 NULL（DWD 兜底，资深数据域）；
        # ⚠️ OSS 部署若维度可空须升 `<=>`（不只「已知限制」— 守护者 Stage 3 注）
        agg_joins.append(f" LEFT JOIN ({agg}) {a} ON " + " AND ".join(f"dim.{d} = {a}.{d}" for d in lf.dimensions))
        sel_metrics.append(f"{a}.{name}")
    driver = "(" + " UNION ".join(union_branches) + ") dim"       # 维度并集（对称不丢）
    sel_dims = ", ".join(f"dim.{d}" for d in lf.dimensions)
    sql = f"SELECT {sel_dims}, {', '.join(sel_metrics)} FROM {driver}" + "".join(agg_joins)
    return _guard(sql + _order_limit(lf))
```

**关键**：① per-metric（每 metric 独立 agg + **自己的 filter** — 守护者 bug 纠正）② 各子查询单 base alias `o`（caliber/dims/filters `o.` 在各自子查询内有效，免 _rewrite_caliber）③ 维度域 UNION 分支 + agg 各注入自己 filters + time ④ 维度列序 = lf.dimensions 序（UNION/agg/ON/outer 一致）⑤ metrics 保序（t{i} by lf.metrics index）。**简化副效**：去掉 by_base 分组逻辑，与 scalar 路径同构。

### 2.3 parser 放宽

`_validate_hit`：去 v0.7.11 的「多 base + 维度 → 拒」pre-guard —— 多 base + 维度交 compiler（共同维度 authoritative，非共同 → raise 回退）：
```python
if "" in objs:
    return False
return True                       # 多 base + 维度由 compiler 共同维度 gate（去 v0.7.11 的 len>1 and dims 拒）
```

`_LOGICFORM_SYS` 铁律：多对象 + 维度放宽为「维度须各引用对象**共同维度**」：
> - metrics 可引用**不同对象**的指标；**多对象 + 维度时，维度须是所有引用对象都有的「共同维度」**（如 city 同时是 orders/users 的维度 → 「各城市 GMV 和 DAU」可编；非共同维度 → 输出 {{"metrics": []}} 回退）。单对象维度可跨对象照常。
