### 2.2 笛卡尔积修复（v0.5.0 commit #1 优先级）

**当前防御不够**：
1. Catalog `RELATIONS` 字典 — gitignored，业务方维护，门槛高极易残缺
2. 双 prompt JOIN 硬约束 — 软约束，LLM 可绕过
3. comma-FROM 守护正则 — 拦不住伪 JOIN（如 `INNER JOIN B ON 1=1`）
4. 22 条 eval cases — 覆盖广度 OK，但 runtime 守护不足

**SQL AST 预校验**（新增 runtime 硬防御）：
- 用 sqlglot 解析生成的 SQL → 抽 JOIN graph
- 检测：≥2 表但无 JOIN ON 条件 / JOIN ON `1=1` 等恒真条件 / 跨表 WHERE 含 ` = ` 但无表前缀
- 触发 → 强制 sql_planner 重生成（带反馈 prompt：「你刚生成了笛卡尔积，请用 RELATIONS 中的 X.id = Y.x_id 修复」）
- 比 prompt 约束硬 / 比 RELATIONS 字典维护成本低 / 代码量 ≤ 200 行

## 3. 重命名 / 重构 / 协议增量提案

### 3.1 bi-agent → knot 重命名（v0.5.0 commit #0）

#### 影响面（实测）
- 含 `bi_agent` / `bi-agent` 字符串的文件：**147 个**
- `bi_agent/` Python 包目录
- `bi_agent/data/bi_agent.db` 用户数据库
- `BIAGENT_MASTER_KEY` env 变量
- `.importlinter` 7 contracts source/forbidden 全部 `bi_agent` → `knot`
- 所有 import / 测试 import / docs 引用

#### 守护者新发现的关键修订 ⭐
**`bi_agent/services/knot/` 与顶层 `knot` 包冲突**：
- 重命名后会变成 `knot/services/knot/orchestrator.py` — 荒谬
- 解决：`services/knot/` → **`services/agents/`**（产品名 knot 留顶层；子目录用通用语义 — 里面就是 3 个 agent）

#### 兼容性（执行者补充，守护者采纳）
- **env 双源**：同时读 `KNOT_MASTER_KEY` + `BIAGENT_MASTER_KEY`，前者优先；旧的发 deprecation log；v1.0 移除
- **DB migration**：startup 检测 `bi_agent.db` 存在 → 自动重命名 `knot.db`（一次性，参 v0.4.5 模式）

#### commit #0 任务清单
```
1. pyproject.toml: name="bi-agent" → name="knot"

5. 全局 sed: from bi_agent.X → from knot.X (含 tests/)
6. env 双源: read KNOT_MASTER_KEY OR BIAGENT_MASTER_KEY  ← 执行者新增
7. DB migration: bi_agent.db → knot.db (startup 一次性)  ← 执行者新增
8. CHANGELOG / CLAUDE.md / docs/plans/* 引用全替换
9. .github/workflows/ci.yml 引用替换
```
