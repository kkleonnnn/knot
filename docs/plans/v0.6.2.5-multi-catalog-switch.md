## §0 一句话目标

让 single-tenant KNOT 支持**多个 catalog 共存 + admin 切换 active catalog**，并保证 **admin 切换瞬间不污染在途 query**（Connection Context 隔离三层 + race 守护）。

**业务动机**：不同业务方（仍在同一 KNOT 实例下，single-tenant）各配自己的 catalog（业务词典 + 表元数据 + RELATIONS）+ 数据源；admin 在它们之间切换，互不串数据。

### D2 — catalog 数据模型迁移方式

**问题**：当前 catalog 存 `app_settings` 4 个 key-value 行（`catalog.tables`/`catalog.lexicon`/`catalog.business_rules`/`catalog.relations`），`catalog.py:reload()` 直读这 4 键，`api/catalog.py` 读写这 4 键。要「多 catalog」需多行。

- **方案 A（执行者推荐）**：新建 `catalogs` 表（id/name/description/tables/lexicon/business_rules/relations/is_active/created_at/updated_at）。`reload()` 改读 `is_active=1` 行。`api/catalog.py` 改读写 active 行。迁移：startup 把现有 4-key 内容 seed 为 `catalogs` 第 1 行（name='默认 Catalog', is_active=1）。app_settings 4 键**保留只读兜底**（防迁移失败数据丢失，v0.7 物理删）。
- **方案 B（低 blast radius 备选）**：保留 app_settings 作「active catalog 运行时内容」源（`reload()` 不改）；新增 `catalogs` 表存**命名 profile**；「切换 active」= 把选中 profile 内容 copy 进 app_settings 4 键 + reload。代价：引入 copy/sync 一致性风险。

**推荐 A**：长期干净、符合「不留技术债」；迁移走 base.py 既有幂等 ADD COLUMN/CREATE TABLE IF NOT EXISTS 模式（先例 `data_sources.http_config` base.py:95 / `totp_recovery_codes` base.py:116）。

### D3 — catalog ↔ data_source 绑定关系

**问题**：R-PB-A1-3「业务方各自配 catalog + 数据源」。切换 active catalog 是否**同时**切数据源连接？
- **方案 A（执行者推荐 — 解耦）**：catalog = 业务元数据（词典/表/RELATIONS）；data_source = 连接。**独立切换**。engine_cache key 维持 `(uid, group_key)`，catalog_id 仅影响 prompt 语义层（LEXICON/BUSINESS_RULES/RELATIONS/TABLES）。最小 blast radius。
- **方案 B（绑定）**：`catalogs` 表加 `default_source_id` 外键，切 catalog 联动切源。engine_cache key 扩 `(uid, catalog_id, group_key)`。

**推荐 A**：roadmap R-PB-A1-2「无跨 catalog 数据隔离」+ A1-5 聚焦 **Connection Context（语义 context）**非物理连接隔离 → catalog 切换主要影响**语义层**，engine 连接仍按 user/source。绑定（B）留 v0.7+ 按需。

### D4 — 2 个 audit_action 命名

roadmap 行 241 字面 = `catalog.switch` / `catalog.context_violation`（新 `catalog.*` 命名空间）。现有约定是 `config.catalog_update`（`config.*` 前缀）。
**执行者推荐**：遵 roadmap 字面 `catalog.switch` / `catalog.context_violation`（LOCKED spec 优先；`catalog.*` 作为「catalog 生命周期」独立语义合理）。仅记录约定差异供 Stage 2/3 知悉。

### D5 — QueryRequest 是否加 catalog_id 参数

**问题**：query 用哪个 catalog？
- **方案 A（执行者推荐 — server 端解析）**：query **不加** catalog_id 参数；server 在 query 入口读当前 `is_active=1` catalog → 捕获进 ContextVar（F5 ①）。前端无需改 QueryRequest。symmetric with 当前「全局 active」语义，只是 active 可切换。
- **方案 B（client 显式）**：QueryRequest 加 `catalog_id`，前端选。破坏当前隐式语义 + 前端 chat 链路改动。

**推荐 A**：single-tenant + admin 切换语义 = server 端 active 解析；F5 ContextVar 捕获的就是「query 入口时刻的 active catalog_id」，天然实现「admin 切换不污染在途 query」。
