## 1. 根因（grounded · 实际代码坐实）

**A（🔴 P0 · 跨问题一+二 · 最危险）—— presenter 失败盲 → 幻觉空集**
- `knot/api/query.py:395-401`：SQL 路径**无条件**调 `run_presenter_step(question, sql, sql_result.rows, …)` —— **只传 rows、不传 `sql_result.error`**。
- SQL 失败时（problem 1 `Unknown database 'futures_admin'` / problem 2 `Unknown column 'market'`）→ `rows=[]` → presenter 无法区分「失败」与「真空」→ 幻觉「无持仓记录」/「本月无数据」。
- query.py:439 发 `error`（红色 banner ✓）+ query.py:440 **同时**发 `insight`（幻觉洞察 ✗）→ 截图实证两者并存。
- **危险**：错误被包装成「看起来合理的空结果」，分析师会信错。
- HTTP **适配器**失败路径（query.py:283-302）已安全 return（友好 error_kind/user_message，不调 presenter）→ **A 只需修 SQL 路径**。

**B（🟠 问题一）—— HTTP 虚拟表 SQL 走 Doris → 原始 Unknown database**
- `pick_http_route` Layer 1 命不中（clarifier 措辞变体）→ 落 SQL 路径 → sql_planner 写 `SELECT … FROM futures_admin.futures_position_list` → Doris/pymysql 连接器 → `Unknown database 'futures_admin'`（原始报错）。
- 换问法命中 → 走对 `GET …/position/list`（同意图时灵时不灵）。
- 已有能力：`catalog.is_http_table(table_full_name)`（catalog.py:184）能判 source_type=http 虚拟表；sqlglot 能从 SQL 提取表名（实证 `futures_admin.futures_position_list`）。

## 2. 范围

**In-scope**：
- A：query.py SQL 路径 presenter **gate 在 `sql_result.error`**（失败跳过 presenter + 不发幻觉 insight；**真空结果[成功+0 行]仍走 presenter**）。
- B：新 helper `http_planner.http_table_in_sql(sql) -> str|None`（sqlglot 提表 + is_http_table）；query.py 失败分支检测到 HTTP 虚拟表引用 → 友好 error_kind=`http_route_miss` + user_message（指引用「当前/实时持仓」表述）替代原始 Unknown database。

**Out-of-scope（资深拍）**：
- C 合约盈亏建模（user_future_pnl 无 market → 用 user_position_pnl）= content 侧 kk 调 metric seed + 导出。

## 3. 红线 R-SL-153~160（守护者 Stage 3 LOCKED · 3 修订已纳入）

- **R-SL-153（A 核心 · 🔴adversarial 复核纠正守护者 Stage 3）**：presenter **仅在 `should_run_presenter(success, error) = success AND not error` 时运行**（= sql_planner success 公式第一子句 `bool(final_sql and not final_error)`，抽 query_steps 纯函数·可单测）；否则跳过 `agent_start`/`run_presenter_step`/`agent_done`（0 LLM + 0 幻觉 insight）。**⚠️ 守护者 Stage 3 的 `not success` 单条件错** —— grounded：`execute_query` 出错返 `([], err)` **永不返 None** → success 公式第二子句 `final_rows is not None` 恒 True → **有 SQL 但执行失败（problem 1/2 Unknown db/column）success=True**，`not success` gate 放行 presenter 继续幻觉（守护者表「DB 错→success=False」事实错，adversarial skeptic+critic grounded 推翻）。也不能只用 `error`（放弃案 error=""）。
- **R-SL-154（A 三场景 · `success and not error` 全对）**：① DB 错（success=True / error set，problem 1/2）→ 跳 ② ReAct 放弃（success=False / error="")→ 跳 ③ 真空成功（success=True / error="")→ 仍走 presenter 说「无数据」（合法）。**单测 `test_should_run_presenter_gate_four_cases` 锁 4 案**（补 critic 指出的「无 A 行为测试」gap，防 `not success` 回归）。失败分支 final `error` 兜底 `fail_msg`（abandon 案 error="" 但需 truthy 让前端 `showErrorBanner=!!(error&&error_kind)` 渲染）。
- **R-SL-155（B 友好拒 · 🟡守护者修订 2）**：失败分支 `http_planner.http_table_in_sql(sql_result.sql)` 命中 → error_kind=**`data_unavailable`**（📡「数据源不可用」**复用既有 7 kind，0 前端改**——`http_route_miss` 不在前端 ERROR_KIND_TITLES 7 key 内会渲染成 ❌「系统错误」）+ user_message 指引「用『当前/实时持仓』表述」；未命中 + 有 DB 错 → **原样透传**（error_kind/user_message 保现状 None，problem 2 Unknown column 不误伤）；放弃无 SQL → `sql_invalid` + 兜底「未能生成有效查询，换个表述」。
- **R-SL-160（失败分支保共享尾 · 🟡守护者修订 3）**：gate **只包 presenter 调用 + insight 字段**；失败分支仍保 `save_message`（db_error 落库）+ `semantic_audit` 写入（语义命中后执行失败仍是 hit，侧表行不能丢，否则命中数漏计）+ `update_user_usage`（clarifier+sql_planner token 已花）+ emit final（前端 ErrorBanner 依赖）。失败时 `insight=""` + 填 error_kind/user_message（对称 R-30/33 成功路径错误字段全 None）。
- **R-SL-156（成功路径 byte-equal）**：gate 只改失败分支；**成功路径（含真空成功）0 行为变更**。
- **R-SL-157（纯 bugfix）**：0 新 schema 表 / 0 新路由 / 0 新 AuditAction（admin 46 / AuditAction 51）；**0 前端改**（复用 data_unavailable）。
- **R-SL-158（版本同步）**：5 源点 0.7.19→0.7.20。
- **R-SL-159（Contract）**：B helper 置 http_planner（已 `import catalog as catalog_loader` L24）；Contract 9 + 8→9 contracts sustained。执行者须核 `is_http_table` 入参格式与 sqlglot 提取的 `db.table` 一致。
