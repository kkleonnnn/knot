### 0.2 根因分层诊断

| 层 | 根因 | 确认度 |
|---|---|---|
| **前端（确定）** | `MetricCard.jsx` 只渲染 `numericCols[0]`（首个数值列）— 复合 metric `SELECT SUM(volume), SUM(deposit)` 2 列 1 行 → 只显第一个，第二个丢弃 | ✅ 代码实证 |
| **clarifier（疑似）** | intent=metric 单标量假设 — 复合 metric 应识别为多值，前端/presenter 需感知 | ⚠️ commit 1 调查确认 |
| **sql_planner（待查）** | 复合 metric 若涉及多表（交易表 + 充值表）→ 可能 fan-out 或漏聚合 → 首列 0 | ⚠️ commit 1 调查确认 |

### 0.3 v0.5.18 部分聚合先例

v0.5.18 R-461 AdminBudgets Hero card 已有"4-stat grid"模式（`repeat(auto-fit, minmax(180px, 1fr))`）— 复合 metric 可复用此设计语言。

## §1 范围（F1-F4）

### F1 — MetricCard 多值网格（commit 1 / 前端确定 bug）

- F1.1 `frontend/src/screens/chat/ResultBlock/MetricCard.jsx`：
  - `numericCols.length === 1`（单值）→ 现有大数字卡片 byte-equal sustained
  - `numericCols.length > 1`（复合）→ **多 stat 网格**（`repeat(auto-fit, minmax(160px, 1fr))` — v0.5.18 R-461 设计语言复用）
  - 每个 stat：值 + 列名 label（mono uppercase）
- F1.2 R-PA-PB-V1 视觉延续：仅依赖 T 25 字段 + 既有设计语言（不扩 buildTheme）

### F2 — clarifier 复合 metric 识别（commit 2 / prompt）

- F2.1 `knot/prompts/clarifier.md`：复合 metric 段
  - 用户一句问 2+ 聚合指标（"X 和 Y" / "X、Y、Z"）→ intent=metric（sustained）+ refined_question 保留全部指标
  - 提示 sql_planner：多指标**优先单表聚合**（`SELECT SUM(a), SUM(b) FROM same_table`）；跨表必先按 grain 预聚合再 JOIN（防 fan-out — v0.4.1.1 R-PB sustained）

### F3 — sql_planner 复合 metric 指引（commit 2 / prompt — 与 F2 同 commit）

- F3.1 `knot/prompts/sql_planner.md`（若存在）或 sql_planner_prompts.py：复合 metric 指引
  - 多指标同表 → 单 SELECT 多聚合列
  - 多指标跨表 → CTE 预聚合后 JOIN（复用既有 fan-out 防御 prompt）
  - **commit 1 调查后确认是否需要**（若 SQL 层无 bug 则 F3 跳过）

### F4 — 守护测试 + docs（commit 3）

- F4.1 `tests/screens/test_metric_card_multi.py` 或前端 snapshot — 多值网格渲染
- F4.2 复合 metric e2e（mock 2 列 1 行 → 网格全显）
- F4.3 4 处版本同步 0.6.2.1 → 0.6.2.2 + CHANGELOG + CLAUDE.md 路线图
