## 1. 决议（范围窄而硬 — 承重坑区域克制）

**一句话**：让**多 base 标量聚合**（如「本月 GMV 和 DAU」，gmv 在 orders 聚合 / dau 在 users 聚合）走确定性编译，而非现在的 R-SL-31 raise 回退 LLM。

### ⭐ 核心架构判断（grounded main `73ca6f8`）—— 两个承重发现

**承重发现 1：v0.7.2 基数膨胀坑的本质是「JOIN 后聚合」；本刀用「标量子查询入 SELECT」按构造免疫。**
- v0.7.2 `_build_multi_object_sql` 是**单 base 聚合 + 维度表 JOIN**（JOIN-后-聚合，靠 `cardinality_safe` n:1 gate 防 base 被乘）。
- **多 base 聚合**（两个 base 都聚合）若简单 JOIN-后-聚合**必膨胀**（这正是 v0.7.2 守护者 MAJOR REVISION 抓的 `_is_fan_out` INNER JOIN 幻觉坑）。
- **本刀形态 = 标量子查询入 SELECT（0 JOIN）**：每个 metric → 独立标量子查询，外层 FROM-less。**0 JOIN → 无 fan-out 可能 → 按构造免疫基数坑**（不需 joingraph / FK / cardinality_safe）。

```sql
-- 「本月 GMV 和 DAU」（gmv@orders + dau@users）
SELECT (SELECT SUM(o.pay_amount) FROM shop.orders o WHERE o.status='paid' AND o.date BETWEEN '...' AND '...') AS gmv,
       (SELECT COUNT(DISTINCT o.user_id) FROM shop.users o WHERE o.date BETWEEN '...' AND '...') AS dau
```
每个标量子查询（GROUP-BY-less 聚合）返**恰 1 行**；外层 FROM-less → 恰 1 行。

**承重发现 2：parser `_validate_hit` 当前拒绝多对象（`len(objs)==1`）→ 多 base LogicForm 根本到不了 compiler。** prompt 铁律 L38 也教 LLM「多对象 → 回退」。**本刀必须 compiler + parser 双改**（否则 compiler 支持也白搭）。

**grounded 安全门验证**（决定形态可行性）：
- `is_cartesian`（`_guard` 跑）：C2/C3/C4 只查每 Select 的 **joins** —— outer 无 join + 子查询单表无 join → 不触发；C1 `_COMMA_FROM_RE` 要求逗号紧跟 FROM 表，SELECT 列表逗号前是 `)` → 不误判。**过** ✓
- `_is_safe_sql`：outer Select 根 + 嵌套 `Subquery` ∈ allowed_roots + 无写节点 → **过** ✓

### In-Scope（标量多 base）
- compiler：多 base 检测 + `_build_multi_base_sql`（标量子查询入 SELECT）
- parser：`_validate_hit` 放宽（允许标量多 base）+ prompt 铁律放宽

### OOS（推后续刀 — 各有承重理由）
- **多 base + 维度**（如「各城市 GMV 和 DAU」）：需「聚合后 JOIN on 共同维度」+ **INNER vs FULL OUTER 正确性承重抉择**（INNER 丢只在一边的维度值 = 静默数据丢失；FULL OUTER 正确但需 COALESCE 维度 + 核 Doris 支持）→ **独立刀**（本刀维度多 base 一律 raise 回退）
- 多 base + 附加 filters / having / window / qualify（单 base grain 概念，标量多 base 不支持 → raise 回退）

## 2. 设计

### 2.1 compiler 多 base 检测 + 分发

`_metric_bases(lf_metrics, metrics_by_name)` —— 存在性校验 + 返回引用 metric 的 base_object 集（排序确定性）；任一未定义 / 无 base → raise。

`_build_sql` 分发加多 base 分支（在单 base 路径**之前**）：
```python
bases = _metric_bases(list(lf.metrics), metrics_by_name)   # 存在性 + base 集
if len(bases) > 1:
    return _build_multi_base_sql(lf, metrics_by_name, tables, time_ctx)   # 多 base 标量
base = bases[0]
# … 单/跨对象维度路径（v0.7.1/.2，byte-equal 不变）
```
（`_resolve_single_object` 重构为 `_metric_bases` + `_build_sql` 内 len 判断；单 base 路径 byte-equal 不变 — 仅 len>1 触发新分支，原 raise 改路由。）

### 2.2 `_build_multi_base_sql`（标量子查询入 SELECT）

```python
def _build_multi_base_sql(lf, metrics_by_name, tables, time_ctx):
    # ⭐ 全矩阵 raise-guard（守护者 Stage 3 — 非 assert）：标量形态 SELECT (subq) AS m, ...（FROM-less 1 行）
    # 无法表达 dims（无组）/ having（无组）/ window（无分区）/ qualify（无 window）/ filters（多 base 归属歧义）。
    # 用 raise CompileError（触发混合架构优雅回退 LLM），非 assert（-O 禁用 + 崩溃非回退）。
    if lf.dimensions or lf.having or lf.window or lf.qualify or lf.filters:
        raise CompileError("多 base 仅标量（维度/having/window/qualify/filters 任一非空 → 回退；聚合后 JOIN 留后续刀）")
    if lf.time and lf.time not in _TIME_KEYS:
        raise CompileError(f"未知 time 枚举 {lf.time!r}")
    subs = []
    for name in lf.metrics:                 # 保序（lf.metrics 序 = SELECT 列序）
        m = metrics_by_name[name]
        physical = _resolve_physical(m["base_object"], tables)   # HTTP/未匹配 → raise
        where = [str(f) for f in _json_list(m.get("filters"))]   # 各 base 自己的口径内置 filters（alias o 不重写）
        if lf.time:
            dc = _resolve_date_col(_json_list(m.get("dimensions")))
            if dc is None:
                raise CompileError(f"metric {name!r} base 无日期列但 lf.time 设定 → 回退")
            start, end = getattr(time_ctx, lf.time)
            where.append(f"o.{dc} BETWEEN '{start}' AND '{end}'")
        sub = f"SELECT {m['caliber']} FROM {physical} o"
        if where:
            sub += " WHERE " + " AND ".join(where)
        subs.append(f"({sub}) AS {name}")
    return _guard("SELECT " + ", ".join(subs) + _order_limit(lf))
```

**关键**：每子查询单 base，alias `o`，caliber/filters 原样（`o.` 前缀在各自子查询内有效，**无需 _rewrite_caliber**）。确定性 —— lf.metrics 保序 + 各子查询纯函数。

### 2.3 parser 放宽（双改第二半）

`_validate_hit`：放宽多对象拒绝 —— 允许**标量多 base**（无维度），多 base + 维度仍拒（compiler 双重防御 R-SL-14）：
```python
if "" in objs: return False
if len(objs) > 1 and lf.dimensions: return False   # 多 base + 维度 → 回退（parser pre-guard）
return True
```

`_LOGICFORM_SYS` 铁律 L38 放宽：
> - metrics 可引用**不同对象**的指标**仅当无维度**（标量多 base 聚合，如「本月 GMV 和 DAU」→ metrics=["gmv","dau"] 无 dimensions，各取标量值）；**多对象 + 维度 → 输出 {"metrics": []}（回退）**。单对象（metrics 同 base）维度可跨对象照常。

### 2.4 抽 multi_base.py 机制（守护者 Stage 3 — 0 循环 import 铁律 · 资深拍）

Stage 2 强制抽 `multi_base.py`（拒主文件肥大）；守护者接受维护性论点**但抓循环 import 雷**：`_resolve_physical`/`_resolve_date_col`/`_json_list`（+ `_order_limit`/`_guard`）**都在 compiler.py**。multi_base import 它们 + compiler import multi_base 分发 = **循环**。**铁律：0 循环 import。**

**grounded 额外约束**：`CompileError` 定义在 compiler.py，被 **5+ 外部模块引用 `compiler.CompileError`**（api/admin/logicform.py ×3 / query_steps.py / monitor_eval.py / 测试）→ 移走会破外部契约或需 re-export。

**三机制（资深拍）**：

| 机制 | 做法 | churn / 评估 |
|---|---|---|
| **C 留 compiler + ACK bump（荐）** | 多 base 路径留 compiler（与单/跨对象并列第 3 build path）；ACK 320→~345 | **0 新模块 / 0 循环风险 / 0 churn**；compiler 是内聚「确定性编译器」，多 base 是合法 build path；守护者**明示 sanctioned fallback**。最高风险刀宜少动件 |
| A DI（守护者次荐）| compiler 预解析 per-metric（physical/where/caliber）→ `multi_base.assemble(resolved, tail)` 纯字符串组装，**0 import compiler** | 0 循环；但 multi_base 仅 ~3 行组装（resolution/guards/time 仍留 compiler）→ 不真瘦身 |
| B 移共享 leaf | 3 helper + `_order_limit`/`_guard` + **CompileError** 移中性 leaf，compiler re-export CompileError | 真瘦身但 churn 最重（CompileError 外部契约 + 5 helper 移动 + re-export）；最高风险刀不宜 |

**执行者建议 C**：CompileError 外部契约 + 最高风险刀「少动件」原则 → 留 compiler + ACK bump 最稳；compiler 真需拆分应另起纯重构刀（不绑承重刀）。**资深拍，我按裁整合 §5/§7。**
