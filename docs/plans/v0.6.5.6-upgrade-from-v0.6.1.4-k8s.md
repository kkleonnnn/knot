## 〇、为什么这次升级特殊（跨 ~40 PATCH 一次性大跨度）

| 维度 | 说明 |
|---|---|
| 🔴 首次引入强制 2FA | 跨 v0.6.2.0 分水岭 → 升级后全员（含 admin）下次登录强制绑定一次（资深拍板：一步到位）|
| 🟡 HTTP 数据源 allowlist | `KNOT_HTTP_ALLOWED_HOSTS` 必须保留（缺则 fail-fast 拒启动）|
| 🟢 界面全换 UI v2 + 新依赖 | pyotp/qrcode/cachetools 随 Dockerfile build 自动装（同 v0.6.1.x `requests` 先例）|

> 功能上**只增不减**：v0.6.1.4 的 HTTP 数据源能力已在 main，且后续 v0.6.1.5~.11 + v0.6.2.1 又加了多项 HTTP 修复与稳态收尾 → 升级是增强，不会回归。

## 一、运维侧需要做的（KNOT 端需求）

**1. 新 image 上线** — tag `0.7.0`（main HEAD）。新依赖 pyotp / qrcode / cachetools 随 Dockerfile build 自动装，无需特殊处理（v0.7.0 起 0 新依赖）。

**2. ConfigMap / env**

| env | 动作 | 说明 |
|---|---|---|
| `KNOT_MASTER_KEY` | 🔴 **保持不变** | 终身密钥；改/丢 → 历史加密的 API key、DB 密码、2FA 秘钥永久不可解 |
| `KNOT_HTTP_ALLOWED_HOSTS` | 🔴 **保留现值** | 当前值含 `landry-futuresadmin-web.landry`（**纯 host 不带 `:8080`**）；现 `futures_admin` 源在线 = 已正确，保留即可 |
| `KNOT_TOTP_REQUIRED` | 不设 | 不设即默认强制（一步到位）|
| `KNOT_TOTP_BYPASS_ADMIN` | 🟢 备用 | admin 自己绑定卡死时临时加 + 重启救援，救完撤；平时不设 |

**3. 数据卷 (PVC) 不变** — `knot.db` 所在 PVC 保持挂载。启动期 `init_db()` 幂等迁移自动加新表/列（catalogs / 2FA totp 列 / audit V2 / OR-only 模型清理 / **v0.7.0 metrics 指标注册表** — `CREATE TABLE IF NOT EXISTS` 自动建，无手动步骤），旧数据不动。

**4. 上线后自动行为**（无需 SRE 操作，知悉即可）— 启动期自动：① 全员旧登录(JWT)一次性失效 ② index.html `Cache-Control: no-cache`（防换 hash bundle 白屏）。

## 二、对在测用户的影响（业务方负责告知）

- 🔴 **首次引入强制 2FA**：升级后所有人（含 admin）下次登录 → 重输账号密码 → **强制绑定 2FA**（Authenticator app 扫 QR + 输 6 位码 + **保存 10 个恢复码**）。
- 界面**全换 UI v2**（功能不变，外观全新）。
- **升级前一句话通知模板**：
  > 系统升级，下次登录需重输账号密码 + 用 Authenticator app（如 Google Authenticator）扫码绑定一次性验证码，请提前装好 app；界面也会更新。

## 四、回滚（SRE 决策）

回退 image tag 到 `v0.6.1.4` + 同 PVC。

- 迁移 additive 幂等，旧版可读新 schema（多出的表/列旧版忽略）。
- ⚠️ 已绑 2FA 的用户数据留库；回滚到无 2FA 的 v0.6.1.4 = 这些用户 2FA 不再校验（安全降级，知悉即可）。
