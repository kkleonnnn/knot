## §0 背景 — 两个精确缺口（执行者实证钉死）

### 缺口 B3（脱敏 V2）：saved_reports 非 admin 脱敏漏配
conversations 历史回放对非 admin 已脱敏（**v0.6.0.17** 移 `sql_text`/`sql` + **v0.6.0.19** desensitize `explanation`/`db_error`）：
```python
# knot/api/conversations.py（基准模式）
if user.get("role") != "admin":
    for m in msgs: m.pop("sql_text", None); m.pop("sql", None)
    desensitize_messages_for_non_admin(msgs, catalog_loader.LEXICON)
```
**但 saved_reports（list / get / run 结果 / export）对非 admin owner 0 同等处理** → 非 admin 分析师查看/导出自己的收藏报表会看到**原始 `sql_text` + `explanation` 中的内部表全名**（如 `dwd_user_deal`）。saved_reports 存 `sql_text`/`question`/`explanation`/`last_run_rows_json`；非 admin owner 经 list/get 拿到原始字段 → 表名泄漏。

> **范围澄清**（实证）：messages export.csv/xlsx 导的是 **rows 数据**（列名 ≠ 表名）→ **不泄漏表名**，不在 B3 范围。B3 精确目标 = saved_reports 非 admin sql_text strip + explanation desensitize（与 conversations 平行）。

### 缺口 B2（audit V2）：5 新 audit_action 未经 v0.4 远古守护者联合复核
v0.6.2.0 TOTP 3（`user.totp.enroll`/`verify_failed`/`reset`）+ v0.6.2.5/2.6 catalog 2（`catalog.switch`/`catalog.context_violation`）= **5 个新 audit_action**（V1 33 → 现 40）。这 5 个**未经 v0.4 远古守护者**（audit_log V1 / R-47 INSERT-only / R-63 messages 排除 / PII 三层防御原始设计者）复核一致性。roadmap-v4 §IV 明列「第 4 次激活 = B2 audit V2 + 脱敏 V2 + TOTP/catalog 5 新 audit_action 联合复核」。

## §2 决策点（执行者推荐 → 待 Stage 2/3 + 资深拍板）

### D1 — saved_reports strip 粒度（sql_text 移除 vs 保留脱敏）
- conversations v0.6.0.17 **完全移除** sql_text（非 admin）。saved_reports 同此 → 非 admin owner 收藏报表不见 sql。
- **推荐**：完全移除 sql_text（与 conversations byte-equal 模式 — 一致性优于差异）。
- **风险**：非 admin owner 收藏自己的报表却看不到 sql（但这是 v0.6.0.17 既定脱敏取舍 — 业务用户不需见内部 schema）。

### D2 — B2 audit V2 补漏深度（仅复核 vs 复核 + 补漏）
- **推荐**：B2.1 联合复核（v0.4 远古守护者）为主 + B2.2 覆盖 gap 实证扫描；若扫描发现未 audit mutation → 补（按 V1 模式）；若 0 gap → B2 = 复核 + 文档化（不强造补丁）。

### D3 — saved_reports run 结果脱敏范围（explanation only vs + rows）
- **推荐**：仅 explanation（rows 是 owner 自己的查询数据 — 不脱敏；与 messages export 同精神 OOS-2）。

### D4 — 规模（是否拆 PATCH）
- **推荐**：**不拆**（B2 + B3 都是聚焦缺口闭合，单 PATCH coherent；B2 多为复核 + 轻补漏）。
