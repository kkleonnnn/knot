## 1. Bug（R-137 grounded 确认 = REAL）

**现象**：`KNOT_SEMANTIC_LAYER=true` 且语义层命中时，顶层 `input_tokens`/`output_tokens`（→ message 行 + 用户用量）**漏计 parse（NL→LogicForm）LLM 的 token**。

**根因**（grounded）：
- `query_steps.run_semantic_compile_step` 命中返回的 `AgentResult` 把 `total_input_tokens`/`total_output_tokens`/`total_cost_usd` **置 0**（parse cost 经 `add_agent_cost` 入 `sql_planner` 桶 R-SL-19）。
- `api/query.py` 顶层汇总 `total_input = clarifier + sql_result.total_input_tokens + presenter`（语义命中时 `sql_result.total_*tokens=0`）→ **漏 parse token**。
- `total_cost` 取 `aggregate_agent_costs(桶)`（含 parse cost）→ **cost 对，token 漏**（不对称）。
- 传导：`save_message(input_tokens=total_input,...)` + `update_user_usage(total_input,...)` 均偏低。

**对比 ReAct 路径**（`run_sql_planner_step`）：其 `AgentResult` **携带** ReAct LLM 的 cost+token（result.total_*），top-level token = 结果和（含之）→ 正确。**语义命中违反此对称**（置 0）= bug 根。

**严重度**：中 —— 成本/预算执行正确（cost 取桶），仅 token 统计/显示偏低；语义层默认 off → 上线/灰度前必修。

## 2. Fix（与 ReAct 对称）

`run_semantic_compile_step` 命中 `AgentResult` 携带 parse cost+token（取代置 0）：
```python
total_cost_usd=parse_res["cost_usd"],
total_input_tokens=parse_res["input_tokens"],
total_output_tokens=parse_res["output_tokens"],
```
- ⟹ query.py 顶层 token 汇总含 parse → message/user-usage token 正确。
- **无双计**：top-level cost 仍取桶（`aggregate_agent_costs`），`AgentResult.total_cost_usd` 不被加进 top-level cost（与 ReAct 同 —— 携带但 top-level 取桶）；`run_semantic_compile_step` 仅 stream 路径调用（grep=2 全 stream），无其他消费者。
