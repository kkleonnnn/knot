## §D 待评审 — Stage 2 评审 7 维度

请独立第三方 AI 按 7 维度 redline（≤ 500 字）：

### Q1 范围完整性
- F1-F7 是否覆盖 TOTP 2FA 强制 enroll 必备所有面（依赖 / Schema / Service / API / Audit / UI / 测试 / Docs）？
- Out-of-Scope（WebAuthn / 多设备 / SMS OTP / enroll skip）推后合理吗？
- 是否漏关键面（如 backup admin / 锁屏会话超时 / TOTP 时钟漂移容忍 ± N 秒）？

### Q2 R-PB-B1-9 R-46-Tx + 应用层 Session 双层防御技术可行性
- SQLite 原生 ACID 事务能否保证 `users.totp_secret` 写入 + `totp_recovery_codes` INSERT 任一失败全回滚？
- "应用层 Session 验证补充防御"（事务 commit 后才标 JWT totp_verified）的实际防御场景是什么？是否存在被绕过的边角情况？
- 如果 commit 之后服务器 crash（JWT 已签发但 Session 未持久化），用户状态如何恢复？

### Q3 R-PB-B1-10 race 100× 守护测试设计合理性
- `asyncio.gather(*[verify(user_id, "wrong") for _ in range(100)])` 误差 ≤ 1 是否过于严格？
- rate_limit 5/min/user 本身已足够防 brute force（约 200 年穷举 1M 组合）—— 100× race 测试是否过度工程？
- 测试在 CI 中可能因 GitHub Actions runner 时钟精度问题不稳定 — 应如何稳定化？

### Q4 R-PB-B1-7 enroll 二次确认 UX
- 要求 2 个连续动态码 — 用户可能困惑"为什么扫码后还要输入两次"
- 业界标准（GitHub / Google）只要求 1 个动态码 — 我们更严格的理由（防扫码错误锁死）是否充分？
- 如果第二个码失败 → secret 已生成但未持久化 → 用户可重新扫吗？UX 是否友好？

### Q5 R-PB-B1-3 admin 自锁防御
- 首位 admin (id=1) 24h 宽限是否过短？业务方部署时 admin 可能没及时绑定 TOTP
- 如何区分"首位 admin"vs"后加的 admin 用户"？仅靠 id=1 是否脆弱（如果 id=1 被删除/降级）？
- 推荐机制：环境变量 `KNOT_TOTP_BYPASS_ADMIN=true` 启动期完全跳过 TOTP 验证 — 但部署 docs 必须明示安全风险

### Q6 Fernet 加密 + 解密性能
- 每次 TOTP verify 都需要解密 `users.totp_secret` — 高频登录场景（如 admin tab 切换）会成为热点
- 是否应缓存解密后 secret（内存 LRU）？缓存时长 / 失效策略？
- 复用 v0.4.5 master_key — 如 master_key rotate 时，TOTP secret 需要同步重加密（v0.5.0 R-74 模式延伸）

### Q7 audit_action 扩展边界
- `user.totp.enroll` / `verify_failed` / `reset` 3 个是否足够？
- 是否漏 `user.totp.verify_success`（成功验证 — 但量大会污染 audit）？
- 是否漏 `user.totp.recovery_code_used`（recovery code 单次使用关键事件）？
