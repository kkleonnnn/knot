## 1. 背景

v0.4.4 收官后，KNOT 进入「准生产」阶段：架构稳 + cost 透明 + 预算可控 + 自纠正可观察 + LLM 真异步 + 错误体验完善。

**当前数据库（`bi_agent/data/bi_agent.db`）以明文存储以下敏感字段**：

| 表 | 字段 | 数据 | 风险 |
|---|---|---|---|
| `users` | `api_key` | 用户级 LLM provider key | 拖库即明文泄露 |
| `users` | `openrouter_api_key` | 用户级 OpenRouter key | 同上 |
| `users` | `embedding_api_key` | 用户级 embedding key | 同上 |
| `users` | `doris_password` | 用户业务库密码 | 拖库 → 业务库被打 |
| `data_sources` | `db_password` | admin 配置的数据源密码 | 同上 |
| `app_settings` | `openrouter_api_key`（key=this 的 value 列） | 全局 OpenRouter key | 同上 |
| `users` | `password_hash` | bcrypt 哈希 | **不需加密**（已不可逆） |

v0.4.5 目标：**至少这 6 类落点全部应用层加密**，不可逆 hash 列保持原状。

## 3. 技术选型

### 加密算法：**Fernet (AES-128-CBC + HMAC-SHA256)**

依据：
- 来自 `cryptography` 库（已是 Python 加密事实标准，`requirements.txt` 添 1 行）
- 自带 IV / authentication tag / 时间戳，使用面无 footgun
- token 格式：`base64-urlsafe(version | timestamp | IV | ciphertext | HMAC)` — 自带前缀语义
- 不选 AES-GCM 直 raw 是因为我们不需要更高吞吐，Fernet 更难用错

### Master key 来源：环境变量 `BIAGENT_MASTER_KEY`

- Fernet 标准 32 字节 base64-urlsafe key
- `.env.example` 加示例 + 生成指令：`python3 -c "from cryptography.fernet import Fernet; print(Fernet.generate_key().decode())"`
- **缺失即 fail-fast**（启动期 `init_db()` 之后立即校验）：抛 `ConfigMissingError`
- 不读 `app_settings`（鸡生蛋问题）

### 加密产物前缀：`enc_v1:` + Fernet token

- 读路径：见 `enc_v1:` 前缀走 decrypt；无前缀视为「未迁移老明文」，由 repositories 层透明回填或直返（迁移期间）
- 写路径：永远带 `enc_v1:` 前缀
- v0.5+ key 轮替时引入 `enc_v2:`，read 走多版本 dispatch，write 切新版本

## 5. 红线候选（待守护者增删改）

| 编号 | 内容 |
|---|---|
| **R-34** | Master key 缺失 → 启动期 fail-fast（抛 `ConfigMissingError`），**严禁** silent fallback 到明文模式或随机生成 key（后者会导致每次重启读不出旧数据） |
| **R-35** | 加密产物**必带** `enc_v1:` 前缀 — 写入永远带；读取见前缀走 decrypt，无前缀视为待迁移老明文（迁移期透明兼容；迁移完后 repositories 单测断言「全部带前缀」） |
| **R-36** | 迁移脚本**幂等**：见 `enc_v1:` 跳过；中断恢复（不依赖外部 lock，靠前缀检查 + 单行事务）；运行多次结果一致 |
| **R-37** | 测试 master key 用 `monkeypatch.setenv("BIAGENT_MASTER_KEY", ...)` 隔离，**严禁** import-time 读取生产 env；conftest 提供 fixture |
| **R-38** | 加密 / 解密**只在 repositories 层** — services / api / models 见到的全部是明文；保持 4 层契约不引新违规（守护当前 import-linter 6 contract KEPT） |
| **R-39** | 前端 API Key 显示 masked + PATCH 空字符串保持原值；**严禁** GET 返回明文（既有路由全部审一遍） |
| **R-40**（候选）| `crypto_service` 模块级 `Fernet` 实例**懒加载** + 进程内单例（避免每次 encrypt 都重建）；测试 fixture 重置实例 |
| **R-41**（候选）| 迁移脚本作为**独立 entrypoint**（`python3 -m bi_agent.scripts.migrate_encrypt_v045`），不进 startup hook（避免每次启动跑一次） |

## 8. 风险点（守护者重点审）

1. **Master key 丢失 = 数据永久不可读** — 文档要红字提示备份，迁移前强制 `.bak` 留底
2. **多进程 / 部署切换 master key 不一致** — 单实例部署目前 OK，集群化后需 KMS（v0.5+）
3. **迁移脚本性能** — 数据量目前 < 1k 行，单事务跑完无压力；万一未来量大需 batch
4. **测试 fixture 与生产 env 隔离** — conftest 必须 setenv 而不是修改全局
5. **R-39 前端 masked** 是否破坏现有 admin 编辑流（编辑器先 GET 再 PATCH 的用户期望需保留）
6. **app_settings 表的 setting key 黑名单**：哪些 key 是敏感的？目前已知 `openrouter_api_key`，需扫一遍实际使用 + 文档化「新增敏感 key 时必须登记」
