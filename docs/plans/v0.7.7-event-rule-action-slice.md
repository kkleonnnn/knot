## §0 一句话

5 层语义已落下 2 层（指标✅ 对象✅）+ LogicForm 治理闭环（七刀）。v0.7.7 **一刀连做上 3 层薄片端到端贯通**：
admin 定义「**指标异动监控**」（事件：metric + 算子 + 基准）→ 规则（阈值/环比 + 动作）→ 点「**立即检查**」→ 命中 → **webhook 通知 + 留痕**。

范式从「被动查询」迈向「主动监控」，但第一版**手动触发**（最方便测，绕调度器卡点）+ **只通知不写回**（不破 read-only 铁律）+ **阈值/异动**（确定性，不做 LLM 归因）。

## §2 决策点（资深拍板项 · 执行者带推荐）

### D1 — scope（资深已拍：vertical slice 三层薄片端到端）
事件定义 + 规则评估 + 动作（webhook 通知/POST）+ 手动「立即检查」+ 留痕，端到端贯通可测。

### D2 — ⭐ schema：统一 `semantic_monitors` 表（事件+规则+动作合一）— ✅ 锁定（资深未异议，按推荐）
- **✅ MVP 合一**：一张 `semantic_monitors`（metric_name + comparator + threshold/baseline_period + time_window + action_type + action_target + enabled + catalog_id）—— 薄片端到端最简。
- 备选：3 张表（events/rules/actions 分层）—— 更贴 5 层模型但 MVP 过重。
- 理由：5 层模型概念上事件←规则←动作分立，但 MVP「指标异动监控」三者天然合一（事件=metric 条件，规则=阈值，动作=响应）；未来需要再拆（append-only 演进）。

### D3 — ⭐ engine 解析（承重 · 同 v0.7.4 D2-A）— ✅ 资深拍 **A**
事件评估取 metric 值需 DB engine。grounded：**catalogs 表无 data_source 绑定** → 无「catalog-correct」engine 可解析。
**A（拍定）= 用触发者（admin）自己的 `get_user_engine`** —— 单租户共享数据源实情下正确（线上所有用户共享 Doris `10.5.184.127` 库 ohx_dwd/ohx_ads）；0 schema 改；engine None / 表不存在 → 该 monitor skip + 留痕「数据源不可用」优雅降级（不中断整批 R-SL-71）。
**诚实登记**：严格 catalog↔数据源绑定 = **v0.7.4 同构 gap**（catalog≠engine）—— 多 catalog 映射不同物理库时 A 不够，根治（选项 C）留独立刀；本刀单租户共享数据源下 A 正确。

### D4 — 事件算子（MVP 阈值 + 环比异动）
- **✅ 阈值**：`metric {>,<,>=,<=,==} value`（如 GMV < 100000）。
- **✅ 环比异动**：`metric 当期 vs 基准期变化 % {>,<} pct`（复用 time_resolver 同比基准；如 GMV 环比跌 >20%）。
- 不做：LLM 归因（卡点3，后续刀）/ 趋势反转 / 多指标组合。

### D5 — 动作（webhook 通知 + 留痕）
- **✅ action_type = `webhook`**：POST action_target URL（title/body JSON）—— 补 `WebhookNotificationAdapter`（实现预留 Protocol）+ **独立 `KNOT_WEBHOOK_ALLOWED_HOSTS` 守护**（守护者 Stage 3 F1 — 复用 is_url_allowed host-only 逻辑但读**独立 env**；严禁混用数据源读取 allowlist `KNOT_HTTP_ALLOWED_HOSTS`，否则污染数据源攻击面）。
- **✅ 留痕**：`monitor_trigger_audit` 侧表（monitor_id/触发时间/metric 值/命中/动作结果）—— append-only。
- **不做**：自动写回业务库（卡点2，破 read-only 铁律，最高风险后续）。

### D6 — eval 0 LLM
事件评估 = 确定性 metric 编译 + 比较，**0 LLM**（无 parser/presenter；阈值/环比是算术）。async endpoint。
