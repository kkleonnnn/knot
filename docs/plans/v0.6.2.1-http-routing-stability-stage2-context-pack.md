## §C 待评审 — Stage 2 评审 5 维度问题

请独立第三方 AI 按 5 维度 redline（≤ 400 字）：

### Q1 C2 三层路由实施可行性
- "正向白名单 catalog source:http + entity 明示" — entity 推断在 KNOT 项目无 LogicForm（推 v0.7+）情况下如何稳健实现？
- "模糊 entity → Clarifier 二次确认" 触发条件如何定义？是否会导致大量正常问题被反复弹确认？
- exclusion regex `(历史|已平仓|强平|爆仓|ADL|(\d+天|\d+月|昨天|上周|N天前))` 中文匹配是否过于脆弱？

### Q2 C1 catalog._load_from_db 推断兜底安全性
- "db 名不在 DataSource 表里 → 强制 source_type='http'" — 是否可能 false positive？
- DataSource 表如果被误删 → 既有 doris 表也会被推断为 http → 危险？
- 推断兜底应该是 fallback 还是替换 admin UI 字段持久化？

### Q3 C3 sql_planner 非 SELECT 拒识边界
- "非 SELECT/WITH 起手" 检测是否覆盖 CTE / UNION / EXPLAIN / 注释开头等场景？
- LLM 输出常含 markdown 代码块标记 ```sql ... ``` — strip 逻辑是否健壮？
- 拒识返"该问题需要外部 API" 错误文案是否会让用户困惑（如果实际是 SQL 表 LLM 临时性失败）？

### Q4 commit 序列合理性
- 5 commits × 各 0.5-1.5 天 = 3.5-4 天指导 — 是否过于乐观？
- commit 2 (C2 三层路由) 是最复杂的 — 是否需要拆 2 commits（路由层 + Clarifier prompt）？
- 测试 commit 4 在最后一个集中跑 — 是否应该跟随业务 commit 分散？

### Q5 e2e 回归测试设计
- "e38de5e76703 类链路回归" — 测试如何模拟生产环境真实数据流？
- 测试是否要包含 admin UI 改 source_type → catalog.reload → pick_http_route 重路由的完整链路？
- 测试边界 case：何谓"模糊 entity"vs"明确 entity"？测试如何 unambiguous 区分？
