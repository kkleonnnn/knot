## 你需要知道的最小背景

- **KNOT** = AI 驱动 BI 助手（自然语言 → SQL → 图表）。React/Vite 前端，纯 inline-style（无 CSS modules），单色 OKLCH 设计系统。
- **Loop Protocol v3** = 强制三阶段评审：执行者出草案 → **你做 Stage 2 初审** → 守护者 Stage 3 终审 → 落地。
- **本 PATCH 性质** = 前端代码重构，**段 3，全项目公认最高风险**。原因：要修改 `frontend/src/Shared.jsx`（视觉 Foundation，18 个屏幕都 import 它），而这是自 v0.5.6 冻结以来**首次受准修改**。过去整个 v0.5.x 系列（约 20 个 PATCH）都有一条铁律 **R-365「Shared.jsx git-diff 必须 = 0」**。本 PATCH 要正式退役这条铁律。
- **目标** = 把散落在 18+ 个屏幕里重复的 inline helper（小组件 / 样式工厂 / 图标）整合进 Shared.jsx，偿还累积 6 个 PATCH 的技术债。拆成 2 个 PATCH：PATCH-1（纯机械搬运）+ PATCH-2（drift 调和）。
- **关键技术约束**：
  - Shared.jsx 现有 12 个 export + buildTheme 返回 26 字段 theme 对象 `T` + 图标库 `I` 含 38 个图标。
  - utils.jsx import Shared.jsx（所以 **Shared 绝不能 import utils，否则循环依赖**）。
  - helper 风格两类：样式工厂 `export const name = (T, ...) => ({...})`（T 第一位置参数）；组件 `export function Name({ T, size, ... })`（T 具名）。
  - 颜色铁律：只能 OKLCH / `color-mix(in oklch, ...)`，禁 hex（少数 boxShadow/overlay 豁免）。

## 你的评审产物（三维度，目标 ≤ 1200 字）

### 维度 A — Redline（破契约 / 命名碰撞 / 隐藏副作用）
逐条检查，给具体行级证据。重点：
- **R-365 退役是否安全**？草案用 R-PB-SH-1「既有 export byte-equal + diff 仅含新增」替代「0 改」。**这个 additive-only 契约有没有漏洞**——能不能既满足它、又悄悄改了既有 export 的行为 / 让未来 drift 溜进来？守护测试该怎么写才挡得住？
- **命名碰撞**：草案把 `Row→FormRow` / `Field→FilterField` / `KV→KVRow` / `inpStyle→inputStyleMono` / SavedReports 本地 `pillBtn→pillBtnCompact`。这些改名是否解决了碰撞？有没有遗漏的碰撞（Shared 已有 export vs 新增）？
- **循环依赖**：草案把 3 个 modal + FeedbackBar 排除出 Shared（因为它们依赖 utils）。这个判断对吗？PATCH-1/PATCH-2 里有没有哪个 helper 实际上也间接依赖 utils 而被误放进 Shared？

### 维度 B — 评分（每项 1-5 分 + 一句理由）
1. PATCH-1 / PATCH-2 **拆分边界**是否正确？（PATCH-1 = 单副本或 md5-byte-equal 机械搬运；PATCH-2 = 需 canonical-body 决策的 drift）
2. **多屏一次性提取铁律 R-PB-SH-5**（一个跨屏 helper 必须在同一 PATCH 改完所有 def + 调用屏，禁单屏阶段化）—— 这是否是防「Shared 与 inline 双真源」的正确防御？有没有更好的做法？
3. **commit 序列**（§5：Shared 加 export 与删 local 同 commit）是否合理？粒度对吗？
4. **红线粒度**：12 条 R-PB-SH 是否过多 / 过少 / 有重叠冗余？
5. **验收标准**（§4）是否可机械验证、无漏网？

### 维度 C — 风险点（作者可能没想到的）
重点追问：
- **探查完整性**：作者承认第一轮探查漏了 3 个屏（critic 抓出），第二轮 md5 复核修正。**你判断还有没有可能漏的 helper / 屏 / 调用点**？（提示：草案是否覆盖了 Chat.jsx / Shell.jsx / Login.jsx / Conversation.jsx 里可能的 inline helper？）
- **Avatar 提取**（PATCH-2）：Avatar 不是命名 helper，是 inline JSX 圆形，跨 3+ 屏，但 `borderRadius:'50%'` 在 8 个文件出现（非全是 avatar）。把 inline JSX 提成组件 + 筛真 avatar 子集，风险多大？建议怎么验证 byte-equal 子集？
- **I icon dict contract-touch**（PATCH-2 加 `I.spark`/`I.flow`，38→40，且 strokeWidth 1.8 vs 库 1.6）：这个时机 / 做法对吗？是否该单独 micro-PATCH？
- **drift 调和的 canonical-body 决策**（PeriodTab 28px vs 30px / TagChip 是否采用 `kind=` superset / statLabelStyle 10.5 vs 11）：每个该怎么定？参数化 vs 选一个 canonical 让另一屏视觉位移，你倾向哪个、为什么？
- **stale 文档**：CLAUDE.md + Shared.jsx header 写「36 icons / 25 字段」，真实源是「38 / 26」。R-PB-SH-11 要求改之前先 grep 确认无守护测试匹配 header 字面。这个前置校验够不够？
