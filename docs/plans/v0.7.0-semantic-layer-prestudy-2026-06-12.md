## 0. 一句话定位

> v0.6.x 把 **NL→SQL 直出**这条路打磨到天花板（"更好的 ChatBI"）；
> v0.7 在它上面叠一层**确定性语义层**，让"已定义"的指标走确定性编译、"未定义"的问题仍走 LLM 兜底 —— **混合架构**，跨入 "Data Agent"。

KNOT 与美团纯确定性指标平台的**根本差异**：美团**先定义后查询**（零定义 = 不能查）；KNOT **零定义可查 + 已定义更准**。LLM NL 入口是卖点不能丢，语义层是兜底不是替代。

## 1. 战略：解锁 README §不能做

| README §不能做 那一列 | 解锁它的语义层 |
|---|---|
| 归因分析（"为什么下降"）| 事件层 + 规则层 |
| 跨业务域聚合（订单×风控×用户）| 对象层（跨域主外键） |
| 数据合理性反检（"这个数对吗"）| 规则层（校验） |
| 动作触发（通知 / 工单 / 审批）| 动作层 |
| 指标口径一致性（同名指标不同人不同算法）| 指标层（口径单一真源） |

前 4 行是**能力跃迁**（做不到 → 做得到）；第 5 行是**质量跃迁**（能做但不可信 → 确定性可信）。**指标层的质量跃迁 ROI 最高、风险最低，应是 v0.7 第一刀。**

## 2. 5 层语义 — KNOT 具体形态

> 自下而上：**指标 ← 对象 ← 事件 ← 规则 ← 动作**。下层是上层的地基；可以只做下层不做上层（增量交付的天然分界）。

### 2.1 指标层（Metric）— v0.7 第一刀

**是什么**：指标 = `(口径公式, 维度集, 血缘)` 的单一真源定义。把"GMV""一次成功率"从"每次 LLM 现写 SQL"变成"查注册表 → 确定性编译"。

**KNOT 数据模型（设想）**：
```
metric:
  name: gmv                      # 唯一标识
  display: 成交额 GMV            # 中文展示
  aliases: [成交额, 交易额, gmv] # → 喂 LEXICON / clarifier
  caliber: SUM(o.pay_amount)     # 口径表达式（基于对象层字段）
  base_object: order             # 挂在哪个对象上（→ 对象层）
  filters: [o.status = 'paid']   # 口径内置过滤（防口径漂移）
  dimensions: [date, city, channel]  # 可下钻维度
  lineage: []                    # 派生指标依赖（如 客单价 = gmv / order_cnt）
  freshness_lag_days: 1          # 复用 time_resolver D-1 默认
```

**衔接现有资产**：
- `LEXICON`（catalog.py:23）已是"术语→表"映射 → 升级为"术语→指标"，metric.aliases 自动喂 LEXICON。
- `BUSINESS_RULES`（自由文本）里那些口径约定 → 结构化进 metric.filters，从"prompt 提示 LLM 别忘了"升级为"编译时强制注入"。
- `freshness_lag_days` 直接复用 `time_resolver.TimeContext`。

**解锁**：口径单一真源（同名指标全站同算法）；派生指标血缘（客单价改了，依赖它的留存率自动跟随）。

### 2.2 对象层（Object / Entity）

**是什么**：业务对象（order / user / risk_event）+ 对象间关系。这是**指标的载体** + **跨域聚合的骨架**。

**KNOT 数据模型**：对象 = `(物理表映射, 主键, 跨对象关系)`。

**衔接现有资产**：
- ⭐ **`RELATIONS`（catalog.py:26）就是对象层的雏形** —— `list[tuple(left_t, left_c, right_t, right_c, semantics)]` 正是主外键关系。v0.4.1.1 建它防笛卡尔积，v0.7 把它**正式升格为对象关系图**。
- `get_relations_for_tables()`（catalog.py:269）按需渲染关系 → 升级为美团式**模型关联**（自动 星型→雪花→星座）。
- `TABLES`（含 source_type=http 虚拟表）→ 对象的物理映射，HTTP 虚拟表让对象可跨 SQL + REST API 数据源。

**解锁**：跨业务域聚合（订单对象 JOIN 风控对象 JOIN 用户对象，沿 RELATIONS 图自动找路径）。

**⚠️ 美团踩坑教训（必吸收）**：不要全量构建星座模型（组合爆炸）。用**半成品星座 + 带阈值 BFS 按需组合**。KNOT 现有 sqlglot AST 笛卡尔积守护（v0.5.1）正好是 BFS 路径合法性的现成校验器。

### 2.3 事件层（Event）

**是什么**：带时间戳的业务事件（下单 / 支付 / 风控拦截 / 退款）+ 事件序列。归因分析的基础。

**衔接现有资产**：
- ⭐ **`time_resolver.py` 是事件层的时间地基** —— TimeContext 已统一时区 / D-1 延迟 / 5 类时间表达 / 同比基准 / 2026 节假日。事件层的"事件发生在哪个时间窗"直接复用。
- time_resolver 注释 L16 已埋伏笔："节假日表 hardcoded（v0.7+ 升级为外部 yaml）"—— v0.7 事件层正好兑现这个承诺。

**解锁**：归因（"GMV 为什么降"→ 拆解到 下单事件↓ / 支付转化↓ / 客单价↓ 哪个环节）。

### 2.4 规则层（Rule）

**是什么**：① 数据合理性校验（"今天 GMV=0 是异常"）；② 业务规则约束（口径红线、阈值告警）。

**衔接现有资产**：
- `budgets` 表 + `budget_service`（v0.4.3）已是"阈值 + 超阈动作"的成熟范式 → 规则层复用这个 (scope, threshold, action) 三件套模型。
- `sql_validator.py`（v0.5.1）已是"SQL 反模式校验器"→ 规则层校验引擎的同构兄弟。

**解锁**：数据合理性反检（结果集回检规则，异常时主动提示"这个数可能不对，因为…"）。

### 2.5 动作层（Action）

**是什么**：查询不止于展示，可触发动作（异常 → 通知 / 工单 / 审批）。这是 "BI Agent → Data Agent" 的临门一脚。

**衔接现有资产**：
- ⭐ **`adapters/notification/base.py`（v0.3.2 占位接口，v0.5.5 删了 lark stub 但保留接口）** —— 动作层的通知通道接口早已预留，等的就是这一天。
- ⭐ **`adapters/http/executor.py`（v0.6.1.4 HTTP adapter）** —— 通用 HTTP 执行器，动作层的"调外部 API 触发工单/审批"现成通道。
- 规则层（2.4）的 action 字段触发动作层执行。

**解锁**：闭环（查询 → 发现异常 → 自动通知负责人 / 建工单）。**风险最高、最该放最后**。

## 3. LogicForm — 结构化中间表示

**核心改造**：当前 `NL → SQL`（LLM 一步直出）；语义层引入 `NL → LogicForm → SQL`（LLM 出结构，语义层确定性编译）。

```
LogicForm（NL 解析产物，结构化语义查询）:
  metrics: [gmv, order_cnt]        # 引用指标层定义（不是现写 SQL）
  dimensions: [city]               # 下钻维度
  filters: [city in (上海, 北京)]
  time: this_month_to_latest       # 引用 time_resolver 枚举
  order_by: [{gmv: desc}]
  limit: 10
```

**为什么是关键**：
- LogicForm 是**确定性的**——同一个 LogicForm 永远编译出同一个 SQL（消除 LLM SQL 抖动）。
- LLM 只负责**好做的事**（NL→结构，语义理解），**难做的事**（口径正确性 / JOIN 路径 / 笛卡尔积）交给确定性编译。
- LogicForm 可被 admin **审计 + 修正**（"AI 理解成了 city，其实你要的是 channel"——改 LogicForm 比改 SQL 友好 10 倍）。

**编译 = 美团三层 SQL 自下而上**：LogicForm → 选对象（路由选表，贪心选最优物理表）→ 沿 RELATIONS 建 JOIN（BFS + 阈值）→ 注入指标口径 + time_resolver 时间窗 → sqlglot 生成 + 笛卡尔积校验 → SQL。

## 4. 混合架构 — 路由决策（KNOT 灵魂）

```
NL 问题
  │
  ├─ clarifier 解析 → 能否映射到已定义指标/对象？
  │
  ├─ 命中（"本月各城市 GMV"，gmv 已定义）
  │     → 语义层确定性路径：NL→LogicForm→编译→SQL  ★高可信
  │
  └─ 未命中（"帮我看下那个奇怪的表里 xxx"，无定义）
        → LLM 兜底路径：当前 sql_planner ReAct  ★零定义卖点保留
```

这正是 v0.6.2.1 已落地的**三层路由思想的延伸**（HTTP vs SQL 路由 → 语义层 vs LLM 路由）。`http_planner.pick_http_route()` 的"正向白名单 / clarifier 二次确认 / exclusion 兜底"模式可直接套用到"语义层命中判定"。

**KNOT 的护城河就在这条岔路口**：纯 LLM BI（不准）和纯语义层 BI（要先定义、门槛高）各占一端，KNOT 站中间——**渐进式确定性**：定义得越多越准，一个没定义也能用。kk 作为唯一数据分析师，可以**按查询热度增量定义**（先定义高频的 10 个指标，覆盖 80% 查询）。

## 7. Open Questions（留给 v0.7.0 Stage 1）

1. **metric 注册表存哪**：复用 `app_settings`（像 catalog 三键）还是新建 `metrics` 表？倾向新表（指标是一等公民 + 需血缘外键 + 需审计）。
2. **LogicForm 是 LLM 直出 JSON 还是 tool-call**：tool-call 更稳（schema 约束）但增 token；JSON 直出省 token 但需校验兜底。倾向 tool-call（与 v0.6.2.1 sql_planner 拒识同精神，确定性优先）。
3. **命中判定阈值**：clarifier 多大置信度才走语义层而非 LLM 兜底？误判进语义层 = 报错；误判进 LLM = 退化为现状（安全）。倾向**保守**（不确定就走 LLM 兜底，宁退化不报错）。
4. **派生指标血缘深度**：限制 BFS 深度防循环依赖（客单价依赖 gmv，gmv 不能反依赖客单价）。复用 audit_service 的 D7 递归深度 3 防嵌套炸弹模式。
5. **跨源对象（SQL 对象 JOIN HTTP 虚拟表对象）**：v0.6.1.4 已禁跨源 JOIN（query.py 路由层守护）。语义层要不要解禁？倾向 v0.7 **暂不解禁**（先把单源语义层做扎实）。
6. **admin.py 拆分前置**：v0.7 要加"指标定义"大模块，admin.py 必须先拆（已在 v0.6.x 段 8 路线图）。**这是 v0.7 的硬前置依赖**。
