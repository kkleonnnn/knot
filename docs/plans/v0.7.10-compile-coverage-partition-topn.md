## 1. 决议（范围窄而硬）

**一句话**：让确定性编译器支持「分区内 top-N」—— 在 v0.7.9 窗口排名列之上**过滤**（如「各地区 GMV 前 3 城市」），补全窗口能力的实用闭环。

**核心判断**（grounded main `47e11dc`）：
- v0.7.9 窗口列只**展示**排名（rank 列），不能**筛选**「前 N」。分区 top-N = 在窗口结果上过滤 `rank <= N`。
- 窗口列在**外层子查询**（`SELECT sub.*, ROW_NUMBER() ... AS rk FROM (inner) sub`），HAVING 在 inner 层 → 窗口列不在 HAVING scope，**不能复用 having**，需独立的「窗口结果过滤」层。
- 标准 SQL 表达 = `QUALIFY`（Snowflake/BigQuery/SQL:2023），但 **Doris QUALIFY 支持不确定** → **保守用外层 WHERE 三层**（`Subquery` 已过 `_is_safe_sql` allowed_roots，grounded doris.py:328）。

**In-Scope**：
- LogicForm 新增 `qualify: list[str]` 字段（alias-based 窗口列过滤谓词，QUALIFY 语义）
- compiler `_finalize` 扩**三层**（递进 byte-equal 守护链）
- parser prompt 教学 qualify（分区 top-N）

**OOS（推后续刀）**：
- 真 `QUALIFY` 关键字（赌 Doris 支持）—— 用外层 WHERE 不赌
- 自定义窗口 frame（`ROWS BETWEEN ...`）
- 多 base 聚合（v0.7.2 基数承重坑）
- CTE/子查询表达

## 2. 设计

### 2.1 LogicForm `qualify` 字段

```python
qualify: list[str] = field(default_factory=list)   # v0.7.10 窗口结果过滤（QUALIFY 语义，alias-based 引 window as_name）
```

`to_canonical_json` —— **空省略键**（末位，window 后）保存量 byte-equal：
```python
if self.qualify:   # R-SL-91：空省略键（末位，window 后）→ 存量 canonical（无 qualify）byte-equal
    obj["qualify"] = list(self.qualify)
```
`from_dict`：`qualify=list(d.get("qualify") or [])`。

### 2.2 compiler `_finalize` 三层（递进 byte-equal）

```python
def _finalize(inner_core, lf):
    if lf.qualify and not lf.window:
        raise CompileError("qualify 需 window 列（无 window 的分区过滤无意义）→ 回退")   # R-SL-94
    inner = inner_core + _having_clause(lf)
    if not lf.window:
        return _guard(inner + _order_limit(lf))                          # 单层 v0.7.1~.8 byte-equal
    cols = ", ".join(_window_col(w) for w in lf.window)
    win = f"SELECT sub.*, {cols} FROM ({inner}) sub"
    if not lf.qualify:
        return _guard(win + _order_limit(lf))                            # 两层 v0.7.9 byte-equal
    qual = " AND ".join(str(q) for q in lf.qualify)                      # 三层（v0.7.10 新）
    return _guard(f"SELECT * FROM ({win}) win WHERE {qual}" + _order_limit(lf))
```

**关键**：qualify 存在时 `win` 两层**不调 `_order_limit`**（移到最外层 L3），与 v0.7.9 F2「inner 不调 _order_limit」同精神 —— `_order_limit` 永远在最终 SELECT 的尾部。

三层 SQL 形态（「各地区 GMV 前 3」）：
```sql
SELECT * FROM (
  SELECT sub.*, ROW_NUMBER() OVER (PARTITION BY o.region ORDER BY gmv DESC) AS rk FROM (
    SELECT o.region, SUM(o.pay_amount) AS gmv FROM shop.orders o GROUP BY o.region
  ) sub
) win WHERE rk <= 3 LIMIT 1000
```

### 2.3 qualify alias-based（同 having R-SL-80 纪律）—— ⚠️ 两个门别混（守护者 Stage 3）

qualify 谓词引 **window 的 as_name + 维度列**（`sub.*` 暴露），**严禁 raw caliber/表前缀**。pass-through `str(q)`（与 having 一致，不重写）。

**alias 正确性的守护点 = parser 死线教学（主防线），不在 `_is_safe_sql`**：
- **门 A `_is_safe_sql`（DQL-only 收口）**：只查**非-DQL**（写节点/多语句/禁根类型）。一个 raw `o.col` 的 qualify 是**合法只读 SELECT** → **过 `_is_safe_sql`** → 但 `o.col` 在 L3 不存在（L3 只见子查询列）→ **execute 时 DB 报 unknown column**（返 db_error，非崩溃，但非干净回退）。`_is_safe_sql` **不守 alias 正确性**。
- **门 B parser 死线教学（alias 正确性主防线）**：parser prompt 强制「严禁 raw o.col/裸 caliber」，从源头不产出 raw qualify。compiler **透传不拒绝**（同 having/filters，无法干净区分 alias vs raw）。
- ⟹ qualify alias 测试 = **parser-prompt-content 断言**（教学含「严禁 raw」，同 v0.7.8 HAVING parser 测试），**非 compiler 拒绝测试**。

### 2.4 parser prompt 教学

JSON 格式加 `"qualify": ["<对窗口列 as_name 的过滤，如 rk <= 3>"]`（window 后）。铁律加：
> **qualify = 分区 top-N**（如「各地区 GMV 前 3」→ window=[{row_number, partition_by:[region], order_by:[gmv desc], as_name:rk}] + qualify=["rk <= 3"]）：⚠️ 必须引 window 的 `as_name`（alias-based），**严禁** raw 口径/表前缀；**无 window 的 qualify 无意义**（编译回退）。无分区 top-N 需求 → 留空 []。
