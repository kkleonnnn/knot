## 1. 总体判断

项目不是普通原型，已经具备较强的工程治理基础：测试覆盖、审计、成本分桶、安全边界、导入约束和部署文档都存在。

当前主要风险不在“能不能跑”，而在以下几类：

1. 语义层（semantic layer）默认关闭，打开前还有计量口径问题。
2. 文档、发布版本和主线代码的状态不一致。
3. 分支和历史 PR 治理不清晰。
4. 流式接口（streaming API）和非流式接口（non-streaming API）能力可能不一致。

## 2. 问题清单

### P1：语义层命中路径存在令牌统计漏计

**现象**

`knot/services/query_steps.py` 中，语义层解析会把成本加入 `sql_planner` 成本桶，但命中后返回的 `AgentResult` 把 `total_input_tokens` 和 `total_output_tokens` 置为 0。

`knot/api/query.py` 在最终保存消息、返回响应、更新用户用量时，又使用 `clarifier + sql_result + presenter` 汇总顶层令牌（token）。

因此，当 `KNOT_SEMANTIC_LAYER=true` 且语义层命中时，可能出现：

- 智能体成本明细（agent costs）里有解析成本；
- 顶层 `input_tokens` / `output_tokens` 偏低；
- 用户用量统计偏低；
- 消息行的顶层令牌统计偏低。

**影响**

目前语义层默认关闭，所以这是上线前风险。一旦灰度或全量开启，会影响成本治理、预算判断和审计口径可信度。

**需要 kk 决策**

语义层是否准备近期打开？如果要打开，建议先修正令牌统计口径，再灰度。

### P1：发布版本与主线版本脱节

**现象**

- 主分支当前为 `v0.7.17`。
- GitHub 最新发布仍是 `v0.6.0.13`。
- README 顶部已写 `v0.7.17`，但发布页仍停在一个多月前的 `v0.6` 系列。

**影响**

外部或团队成员如果按发布页部署，会拿到明显落后的版本。项目真实状态需要从主分支判断，但这不是普通用户会自然理解的规则。

**需要 kk 决策**

选择一种发布策略：

1. 继续维护正式发布：补 `v0.7.17` 或下一个稳定版本发布。
2. 改为主分支即事实版本源：README 明确声明发布页暂不作为部署入口。

### P2：README 产品边界和技术状态过期

**现象**

README 同时存在两套叙事：

- 顶部写当前版本 `v0.7.17`，并列出大量语义层能力。
- 后文仍写“v0.6.x 阶段宣告”，并把部分 `v0.7+` 能力写成未来范围。
- 技术栈部分仍写 `v0.6.4.1`、旧路由数、旧导入边界数量和旧测试描述。

**影响**

产品边界不清，会让读者误判项目阶段：到底是“更好的聊天式商业智能（ChatBI）”，还是已经进入“数据智能体（Data Agent）”阶段。

**需要 kk 决策**

README 是否要重写为当前真实状态：

- 当前已稳定支持什么；
- 当前实验性支持什么；
- 当前明确不支持什么；
- 语义层默认关闭时，对用户意味着什么。

### P2：非流式接口与流式接口能力不一致

**现象**

语义层编译步骤只在 `/api/conversations/{conv_id}/query-stream` 路径中出现。

非流式 `/api/conversations/{conv_id}/query` 仍走旧的同步查询链路，没有接入语义层。

**影响**

如果前端和正式产品只使用流式接口，这可以接受。但如果脚本、外部调用方或测试环境仍依赖非流式接口，就会出现同一个问题在两个接口上得到不同 SQL 生成路径的情况。

**需要 kk 决策**

非流式接口属于：

1. 正式支持接口：需要补齐语义层能力。
2. 兼容保留接口：需要文档注明能力落后。
3. 待废弃接口：需要设定迁移和删除计划。

---

### P3：前端构建包偏大

**现象**

前端构建通过，但主 JavaScript 包约 `1.56 MB`，压缩后约 `484 KB`。

**影响**

内部工具场景通常可接受。若后续面向更多用户、弱网环境或外部部署，首屏加载和缓存更新成本会变高。

**需要 kk 决策**

是否现在处理代码拆分（code splitting）。建议如果仍是内部工具，可暂缓；如果准备扩大使用范围，应把路由级拆包和图表依赖拆分列入前端治理任务。

---

### P3：依赖版本缺少更强约束

**现象**

后端测试本地通过，但测试过程中出现 `httpx` 相关弃用警告。当前依赖版本范围较宽，未来上游版本变化可能导致测试或运行行为漂移。

**影响**

不是当前阻断项，但会增加发布可重复性风险。

**需要 kk 决策**

1. 是否在正式发布前增加约束文件（constraints file）或锁定关键依赖范围。
2. 是否把弃用警告转成后续维护任务。
