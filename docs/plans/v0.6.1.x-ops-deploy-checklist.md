## 二、部署步骤（按顺序执行）

### Step 1. 备份 DB（保守起见）

```bash
cp knot/data/knot.db knot/data/knot.db.bak-pre-0.6.1.4-$(date +%Y%m%d-%H%M%S)
```

### Step 2. 拉新代码

```bash
git fetch origin
git checkout main
git pull origin main
git log -1  # 验证 HEAD 显 330a117 v0.6.1.10
```

### Step 3. 装新依赖

```bash
pip install -r requirements.txt
# 关键新增：requests>=2.31.0（HTTP adapter 依赖）
```

### Step 4. 加 env 变量

在生产 `.env` 末尾加：

```
KNOT_HTTP_ALLOWED_HOSTS=futuresadmin.0p.oh
```

> ⚠️ 此 env **必填**。catalog 含 HTTP 虚拟表时，未设此变量 KNOT 会 fail-fast 拒启动（R-PB2-3 安全守护）。
>
> 多 host 用逗号分隔：`futuresadmin.0p.oh,otherapi.example.com`
>
> 如走 k8s 内部 service 域名：替换为内部 DNS（如 `futuresadmin.default.svc.cluster.local`）

### Step 5. 检查 KNOT_MASTER_KEY

如旧 `.env` 还在用 `BIAGENT_MASTER_KEY`，需要加同值的 `KNOT_MASTER_KEY`：

```
KNOT_MASTER_KEY=<复制 BIAGENT_MASTER_KEY 的值>
```

（v0.6.0 起单源；BIAGENT 兼容已撤回）

### Step 6. 部署私货 `_local_catalog.py`

通过**安全渠道**（1Password / 加密邮件 / 内部文件分享）收到 kk 发的 `_local_catalog.py` 后：

```bash
# 校验 md5（防文件损坏）
md5sum _local_catalog.py
# 期望输出: bbc679326df2075e9b5d26622fafbe09  _local_catalog.py
#（如不一致立即叫 kk 重发）

# 放到正确路径
cp _local_catalog.py knot/services/agents/_local_catalog.py
chmod 644 knot/services/agents/_local_catalog.py
```

> ℹ️ 此文件含部署方业务方言（持仓表 schema 声明 + lexicon），按设计 **gitignored** 不进 OSS 仓。

### Step 7. 重建前端

```bash
cd frontend
npm install
npm run build  # 输出到 ../knot/static/assets/index-*.js
```

### Step 8. DB 自动迁移（无需手动）

启动期自动执行 `ALTER TABLE data_sources ADD COLUMN http_config TEXT DEFAULT ''`
（幂等，已加过不再加）

### Step 9. 重启 uvicorn

```bash
# 现有方式重启即可
systemctl restart knot.service     # 如果用 systemd
# 或 docker compose restart knot     # 如果用 docker compose
# 或 kill PID + 重新 nohup ...        # 朴素重启
```

### Step 10. 启动验证

```bash
# 1. 版本号
curl http://localhost:8000/openapi.json | jq '.info.version'
# 期望: "0.6.1.4"

# 2. 路由数
curl http://localhost:8000/openapi.json | jq '.paths | keys | length'
# 期望: 60+

# 3. uvicorn 日志看启动 banner
tail -20 /var/log/knot/uvicorn.log  # 或对应 log 路径
# 期望含: "KNOT_MASTER_KEY 已加载（Fernet）" + "Application startup complete"
```

## 三、部署后 admin（kk）侧操作

运维完成 Step 1~10 后，**通知 kk**。kk 接下来 5 分钟做：

### A. 登 prod admin UI 加 futures_admin 数据源

1. 浏览器开 `https://knot.0p.oh/`（或 prod 实际域名）
2. admin 账号登录
3. 数据源页 → `+ 添加数据源`
4. 选 `HTTP API (REST)` 类型
5. 填表单：
   - 名称：`futures_admin`
   - 说明：`撮合持仓 admin API`
   - Base URL：`http://futuresadmin.0p.oh`（或 k8s 内部域名）
   - Auth Header 字段名：`key`
   - Auth Header 值：`<production token>`
   - Timeout：`5`
6. 保存后**记录 data source 自增 id**（截图或抄一下）

### B. 若 prod data source id ≠ 3 → 改 `_local_catalog.py` 重传

```bash
# 本地改 _local_catalog.py
# 找到两处 "source_id": 3,
# 改为实际 prod id

# 重算 md5
md5sum _local_catalog.py
# 把新 md5 + 新文件发给运维

# 运维替换 + uvicorn auto-reload 自动 pickup
```

## 四、烟雾测试（部署后 30 分钟内做完）

### 运维侧（基础健康检查）

```bash
# 1. version
curl -s http://localhost:8000/openapi.json | jq '.info.version'

# 2. 登录（用 admin/admin123 或新设密码）
TOKEN=$(curl -s -X POST http://localhost:8000/api/auth/login \
  -H "Content-Type: application/json" \
  -d '{"username":"admin","password":"<admin pw>"}' | jq -r .token)
echo "Token len: ${#TOKEN}"  # > 100

# 3. 数据源列表通（应 2 条 — doris + futures_admin）
curl -s http://localhost:8000/api/admin/datasources \
  -H "Authorization: Bearer $TOKEN" | jq 'length'

# 4. HTTP 数据源探针 online
curl -s http://localhost:8000/api/admin/datasources \
  -H "Authorization: Bearer $TOKEN" | \
  jq '.[] | select(.db_type == "http") | .status'
# 期望: "online"
```

### kk + 业务方侧（功能验证）

| # | 测试场景 | 期望表现 |
|---|---|---|
| 1 | 登 https://knot.0p.oh/ | Login → Home 屏正常 |
| 2 | 数据源页 | 2 行 ● 正常（doris + futures_admin） |
| 3 | 新对话提问：`今日合约交易量` | doris 路径，返数字（如 12k+ USDT） |
| 4 | 新对话提问：`用户 [真实 prod user_id] 当前 BTC 持仓` | HTTP 路径，4 步 K/N/O/T 全 ✓，返表格 |
| 5 | 时间字段显示 | 应为 `YYYY-MM-DD HH:MM:SS UTC` 字符串（非 Unix 数字） |
| 6 | 表格列名 | 中文（如 "更新时间 / 未实现盈亏 / 开仓价 / 市场"） |
| 7 | Trace step 文案 | `本次结果由外部 HTTP API 返回，→ 1 次 GET 请求。`（**不含 endpoint path**） |
| 8 | 不提示 path 暴露 | error / SQL 框 / Knowledge 都不出现 `/admin/api/...` 路径 |

## 六、回滚预案

### 代码回滚

```bash
git checkout 2ee7efb  # v0.6.0.27
cd frontend && npm run build
systemctl restart knot.service
```

### DB 不需回滚
- 新加的 `http_config` 列旧版直接忽略
- 旧版本无 HTTP 表 / 无 HTTP 查询路径，所以新数据 (futures_admin source 行) 也不会被旧代码使用

### `.env` 回滚（可选）
- 删 `KNOT_HTTP_ALLOWED_HOSTS=...` 这行（不删也不影响）

### 已知 v0.6.1.4~1.10 风险点
- ✅ HTTP 路由失败 fail-soft → 友好 ErrorBanner，不 crash 整应用
- ✅ SQL 路径 (doris) 0 改动，原查询全部仍工作
- ✅ DB schema 兼容（新增列不破旧版）
- ⚠️ 仅在用户触发 HTTP 路由时才走新代码（如问"持仓"才有风险）
- ⚠️ 业务方 0t.oh/0p.oh API 抖动 → KNOT 显友好错误（R-PB2-5）
