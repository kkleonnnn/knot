## 9. Stage 3 终审整合（✅ LOCKED 2026-06-22 · 权威终态）

> 守护者 Stage 3 = **ACCEPT WITH MAJOR REVISION**（挖到 Stage 1+Stage 2 双方承重盲区）。资深拍板岔口 **(a) 加基数**。**本节与本体早期措辞冲突处以本节为准。**

### 🔴 承重盲区（守护者发现 + 执行者 R-137 独立核验全坐实）
`_is_fan_out`（sql_planner_tools:86）的 `_LEFT_JOIN_NAMED_TABLE_RE` **只匹配 LEFT JOIN** → D5 INNER JOIN 产 0 LEFT → **恒返 (False,"") 结构性永不触发** = 幻觉守护（Stage 1 + Stage 2 双漏，未读实现）。而 INNER JOIN 1:N fan-out（`SUM(o.amt)` × 明细行膨胀）是真实风险，`is_cartesian` 也不管（合法 ON 的 1:N 照过）。RELATIONS `semantics`（5th）= 自由文本无基数（且 len 可 4）→ 编译器无数据判定 join 是否膨胀 → 拿不出真守护。指标注册表卖点 = 确定性+可信，静默膨胀 SUM 比 LLM 更坏。

### 岔口 (a) LOCKED — RELATIONS 加基数 → 真·确定性 fan-out 守护
- **RELATIONS 加第 6 元素 cardinality**：`n:1` / `1:1` / `1:n` / `n:n`（缺省 `unknown`；**向后兼容 len≥4** — 4=keys / 5=+semantics / 6=+cardinality）。数据层 = catalog relations JSON（无 DB 迁移）。
- **编译器仅沿 `n:1` / `1:1` 边跨对象聚合**（聚合对象在「1」侧，行不膨胀）；`1:n` / `n:n` / `unknown` / 无基数 → **回退 LLM**（R-SL-31）。这是**真守护**（声明基数拒不安全 join），替代幻觉 `_is_fan_out`。
- **admin UI cardinality widget 推迟**（admin 暂经既有 catalog RELATIONS JSON 编辑器设第 6 元素；专用 widget = 未来 polish）→ v0.7.2 后端聚焦。渐进式确定性：kk 按需声明基数，未声明→LLM 兜底安全。

### 新增/修订红线
- **R-SL-23（修订）**：BFS 确定性枚举（邻接按表名排序）判定路径唯一性；同输入 → 同路径。
- **R-SL-27（弃用）**：原 `_is_fan_out` fan-out 守护否决（INNER 幻觉）→ 由 R-SL-31 取代。
- **R-SL-30（新 · 非-caliber `o.` 归属）**：`_build_sql` 给 dimensions（SELECT+GROUP BY）+ date_col 也加 alias 前缀；多对象时按**成员归属唯一对象**用其 alias；**0 或 ≥2 拥有 → 歧义 → 回退 None**。
- **R-SL-31（新 · 跨对象 fan-out 诚实守护 · C3 实现期精化收窄，资深 2026-06-22 拍）**：执行者建 C3 时下钻发现基数 gate「仅 n:1/1:1 边」**必要但不充分** —— 反例 `SUM(orders.amt)+COUNT(users.id)` over orders→users n:1：基表 orders 不乘（对），但 JOIN 对象 users 的非-DISTINCT 聚合仍按订单行数膨胀（users 数成订单数）。**精确安全范围收窄**：① 所有聚合 metric **共享单一 base 对象**（grain）② 其他对象经 **n:1/1:1 边 joined（在「1」侧，base 不乘）只供维度/过滤不聚合** ③ **多 base 聚合 / 任何 1:n·n:n·unknown 边 → 回退**。交付价值 = **按跨对象维度切单 base 指标**（最常见跨表 BI）；多对象聚合（需 DISTINCT/pre-agg）留未来。**严禁静默膨胀**。守护测试：多 base 聚合 / 1:n 边 → 回退。
