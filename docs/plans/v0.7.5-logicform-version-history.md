## §2 决策点（资深拍板项 · 执行者带推荐）

### D1 — 范围（推荐：read-only 窄）
- **✅ 推荐 = 版本历史 list + diff 视图（read-only）**。
- **推迟 v0.7.6+**：① 回滚/恢复某版本（= mutation，新 AuditAction `logicform.rollback` + 把旧版本设为 active；需 active-version 语义设计）② 事件/规则/动作层。
- 理由：read-only 零 mutation 零风险；回滚引入「哪个版本是 active 口径」语义（侧表当前无 active 标记），是独立设计，单独一刀。

### D2 — diff 算在哪（推荐：前端）
- **✅ 前端**：LogicForm 是小 dict，6 字段 field-level diff（增/删/改）是 trivial JS；SQL 文本 diff 前端逐行/字符对比。
- 后端**仅**每版本重编译 SQL（compile_logicform 在后端）。0 后端 diff 复杂度。

### D3 — 每版本 SQL 来源（守护者 Stage 3 分层 · 保真度）
⚠️ **保真度承重点**：`compile_logicform` 用**当前** metric 注册表（list_metrics 读当前）+ catalog → 重编译 SQL 是「**现在**编成什么」非「**当年**实跑」。**LogicForm（存的 canonical_json）才是忠实历史源**；SQL 仅当前渲染。分层（R-SL-51/56）：
- **near-miss**（compile_error_reason 非空）→ **显存的 reason 不重编译**（历史真相；重编译用当前 metric 会复现**可能不同**的 error，甚至现成功 → 失真 + 省一次注定失败编译）。
- **hit**（reason 空）→ 重编译取 SQL + **UI 标注「当前重编译」非原始 SQL**；现失败（metric 漂移/删）→ try-except → 显「当前编译失败（口径可能已变更）」**与历史 near-miss 区分**。
- **diff 主体在 LogicForm（忠实）**；SQL 是分层辅助展示（diff 忠实性 ⊥ SQL 当前性，不互污染）。

### D4 — ⭐ AdminLogicForm cap（推荐：拆子组件）
- AdminLogicForm 196/200 近满 → 版本历史 UI 必超。
- **✅ 推荐拆子组件** `LogicFormHistory.jsx`（drawer 内嵌；镜像 v0.6.x C5「screens/<family>/」子组件拆分先例 — 如 `screens/audit/`）。AdminLogicForm 保编排 + 引子组件，回落 ≤ 200。
- 备选（不推荐）：提 cap —— 单文件超 200 损可读性。

### D5 — audit（推荐：read-only 不新增）
- **✅ 版本历史/diff 是 read-only**（看演化）→ **不产生新 AuditAction**（停 46）；不写 message/侧表。
- 回滚（mutation）才需新 audit → 留 v0.7.6。
