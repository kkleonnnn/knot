## §2 决策点（资深拍板项 · 执行者带推荐）

### D1 — scope（推荐：HAVING only）
- **✅ 推荐 v0.7.8 = HAVING 单项**（聚合后过滤）。窗口函数（排名/同环比）/ 多 base 聚合（v0.7.2 承重坑 基数膨胀）/ CTE/子查询 = **风险递增的后续独立刀**（逐项放，vertical-slice 哲学）。
- 理由：HAVING 单子句、低风险、高频、纯 SQL；与 filters 同范式（已验证）。

### D2 — HAVING 表达式形态（守护者 Stage 3：**强制 alias-based**）
- **✅ `having: list[str]` 强制引 metric alias**（如 `["gmv > 10000"]` 引 SELECT `AS gmv`）—— 镜像 lf.filters 原始进子句范式，但**强制 alias-based 非软建议**（parser enforce + 测试）。
- ⚠️ **多对象非对称**（守护者坐实 `_build_multi_object_sql` L31-34 lf.filters 原始不重写）：alias-based HAVING（`gmv`）单/多对象都对（HAVING 引 SELECT alias）；**原始 `o.col`/裸 caliber 多对象 ✗**（`o.→t0` 重写后 raw `o.` 过 _is_safe_sql 但 DB 报未知列 = 非干净回退）→ **严禁**。HAVING 天然 alias 形态使强制 alias 干净可达。
- 安全：注入 `;`/DML → execute_query 顶 `_is_safe_sql`（单语句/DQL-only）拒（同 filters，已验 v0.7.4 R-SL-43）。

### D3 — HAVING 是否要求 GROUP BY（推荐：不强制）
- **✅ HAVING 独立于 GROUP BY**：有 dimensions（GROUP BY）→ HAVING 过滤分组；无 dimensions → HAVING 过滤单聚合行（合法 SQL）。编译：HAVING 子句在 GROUP BY 之后（无 GROUP BY 则在 WHERE 之后）。
- 不额外 raise（HAVING 表达式合法性由 DB 判 + _is_safe_sql 守护）。

### D4 — byte-equal 守护（推荐：无 having 时现状不变）
- **✅ lf.having 空 → SQL byte-equal v0.7.1~.7**（R-SL-78）—— 现有查询 0 漂移；HAVING 仅在 having 非空时出现。
