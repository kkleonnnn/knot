## §0 背景 — 结构半留下的"记录但不生效"缺口

v0.6.2.5 结构半落地：catalogs 表（多行）+ `users.active_catalog_id`（per-user）+ `POST /api/catalog/switch`（记录切换）+ 选择器 UI。**但 query 仍用全局默认 catalog**：

- `catalog_loader` 的 `TABLES / LEXICON / RELATIONS / BUSINESS_RULES` 是**模块级全局**，`reload()` 从 catalog id=1 加载一次（结构半 commit 3）。
- SQL planner / schema_filter / presenter 直接读 `catalog_loader.TABLES` 等全局 → **所有用户共享 catalog id=1**，`active_catalog_id` 切换被记录但**不影响 query**。
- 结构半 UI 已诚实标注「per-user 切换 · query 生效 v0.6.2.6」。

**并发半目标**：query 时按**当前用户的 active_catalog_id** 解析 catalog 内容（catalog_repo.get_active_catalog 已就绪），让多 catalog 真正影响查询 —— 且**并发安全**（用户 A 的 query 不被用户 B 的 active 切换污染）。

**核心难点**：全局态 → 请求态。模块级全局无法 per-user-per-request。需 ContextVar 把"当前请求的 active catalog"绑定到 async 上下文，catalog 访问点改读 ContextVar。

## §2 决策点（执行者推荐 → 待 Stage 2/3 + 资深拍板）

### ⭐ D1 — ContextVar 回退语义（全局 fallback vs 强制 set）

- **现状**：catalog 访问点直读全局 `TABLES/LEXICON`。非 query 路径（admin reload / startup / http_planner）也读全局。
- **推荐**：ContextVar **未 set 时回退模块全局**（`current_catalog()` = ContextVar.get() or 全局）。query 路径 set ContextVar（per-user）；非 query 路径不 set → 用全局（向后兼容 0 行为变更）。
- **风险**：回退掩盖"忘记 set"的 bug。缓解：第②层 assert 在 SQL 执行前强校验 query 路径必有 ContextVar。

### ⭐ D2 — catalog 访问点改造范围（哪些读全局的点要 ContextVar 化）

- **现状**：`catalog_loader.TABLES/LEXICON/RELATIONS/BUSINESS_RULES` 全局被多处直读（sql_planner / schema_filter / presenter / http_planner）。
- **推荐**：引入 `catalog_loader.current_catalog()` 返回 dict（ContextVar 优先 + 全局回退）；**仅 query 链路**读取点改用（sql_planner schema 注入 + presenter business_rules 注入）。http_planner / admin reload **不改**（OOS / 非 per-user）。
- **待 Stage 2/3 划定**：精确改造点清单（防漏 / 防过度改造破其他链路 byte-equal）。

### ⭐ D3 — query.py Extract Method 剥离对象（腾 12 行余量）

- **现状**：query.py 428/440（12 行）。F2 入口捕获 + F4 assert 会撞 440。
- **推荐**：ContextVar 捕获 + assert 逻辑剥离到 `query_steps.py`（57 行余量）或新建 `query_helper.py`；query.py 主控只调用（保 SSE 样板 R-94 精神）。
- **待 Stage 2/3 划定**：剥离边界（哪些行移走）+ 是否新建文件 vs 复用 query_steps。

### D4 — assert 失败处置（raise vs 降级 + audit）

- **推荐**：第②层 assert 失败 = 隔离漏洞（async race）→ **raise + context_violation audit**（不静默服务错 catalog）；与结构半兜底熔断 ε2 fail-fast 同精神。
- **待 Stage 2/3**：raise 的用户可见错误文案 + 是否 retry。

### D5 — 规模（是否拆 PATCH）

- **推荐**：**不拆**（并发半本身已是 D1 拆分的产物；F1-F8 是一个 coherent 并发隔离单元）。query.py Extract Method 是配套瘦身非独立主题。
