## 1. 决议（CTE 落到「结果再聚合」· 窄而硬）

**一句话**：让「对编译结果再聚合一层」（如「GMV 超 1 万的**城市数**」count groups、「top-5 城市 GMV **合计**」sum）走确定性编译 —— `WITH r AS (<现有编译 SQL>) SELECT COUNT(*)/SUM(col) FROM r`。

### ⭐ 核心判断（grounded main `7b11aed` + AskUserQuestion 拍板）

CTE 是**结构非能力**，落点经 grounded 决策：
- **窗口已能全局 `OVER ()`**（`_window_col` 空 partition_by → `OVER ()`）→ vs-total 部分已覆盖。
- **metric 是原子 caliber**（`SUM(o.pay_amount)` 单表达式，无派生算术）→ **占比/人均 = metric÷metric 需扩 metric 模型**（派生 caliber，超「编译覆盖」范畴，资深拍**不在本刀**，留独立「派生指标」模型刀评估）。
- **两级聚合 grain-change** 复杂（LogicForm 须表达两级 grain）→ 留后续。
- **资深拍「结果再聚合」**：`WITH r AS (<现有编译>) SELECT <外层聚合> FROM r` —— **复用全部现有编译覆盖作 body** + 外层标量聚合。最窄、原子模型兼容、**基数 by-construction 安全**（外层标量 1 行）。

```sql
-- 「GMV 超 1 万的城市数」（复用 v0.7.8 HAVING 作 body）
WITH r AS (
  SELECT o.city, SUM(o.pay_amount) AS gmv FROM shop.orders o GROUP BY o.city HAVING gmv > 10000
) SELECT COUNT(*) AS result FROM r

-- 「top-5 城市 GMV 合计」（复用 v0.7.10 分区 top-N 作 body）
WITH r AS (<分区 top-N SQL，输出 city/gmv/rk + qualify rk<=5>) SELECT SUM(gmv) AS result FROM r
```

### In-Scope
- LogicForm +`outer` 字段（外层聚合）
- compiler `compile_logicform` outer-aggregate wrap（复用 `_build_sql` 作 CTE body）
- parser prompt 教 outer

### OOS（推后续刀 / 资深裁）
- 派生指标/占比（metric÷metric → 需扩 metric 模型，资深裁不在本刀）
- 两级聚合 grain-change（先细粒度→再粗粒度 GROUP BY，复杂）
- 纯语法 CTE 改写（把现有嵌套子查询改 WITH = 0 新能力，无价值）

## 2. 设计

### 2.1 LogicForm `outer` 字段

```python
outer: dict = field(default_factory=dict)   # v0.7.14 外层聚合 {func:count|sum|avg|min|max, arg?:列}
```
`to_canonical_json` 空省略键（**末位，qualify 后**）：
```python
if self.outer:   # R-SL-118：空省略键（末位）→ 存量 canonical byte-equal
    obj["outer"] = dict(sorted(self.outer.items()))
```
`from_dict`：`outer=dict(d.get("outer") or {})`。

### 2.2 compiler outer-aggregate wrap

`compile_logicform`（唯一 outer reader；`_build_sql` 不读 lf.outer → 0 递归）：
```python
def compile_logicform(lf, catalog, time_ctx):
    ...
    inner_lf = lf
    if lf.outer and not lf.limit:        # R-SL-122：outer 聚合全部（无显式 top-N）→ inner 无 LIMIT（哨兵 -1）
        inner_lf = replace(lf, limit=-1)
    inner = _build_sql(inner_lf, metrics_by_name, tables, time_ctx, relations)   # 复用全部现有覆盖（忽略 lf.outer）
    if lf.outer:
        return _guard(f"WITH r AS ({inner}) SELECT {_outer_expr(lf)} FROM r")   # CTE 外层聚合
    return inner   # outer 空 → 无 wrap，存量 byte-equal
```

`_outer_expr(lf)`（func 白名单 + arg alias-based）：
```python
_OUTER_FUNCS = {"count", "sum", "avg", "min", "max"}
def _outer_expr(lf) -> str:
    func = str(lf.outer.get("func", "")).lower()
    if func not in _OUTER_FUNCS:
        raise CompileError(f"未知 outer 聚合 {func!r}（白名单 {sorted(_OUTER_FUNCS)}）→ 回退")
    arg = str(lf.outer.get("arg") or "")
    if func == "count" and not arg:
        return "COUNT(*) AS result"
    if not arg:
        raise CompileError(f"outer {func} 需 arg → 回退")
    if arg not in set(lf.metrics) | set(lf.dimensions):   # alias-based：∈ inner 输出列（metric/维度）；严禁 raw
        raise CompileError(f"outer arg {arg!r} ∉ inner 列（metric/维度）→ 回退")
    return f"{func.upper()}({arg}) AS result"
```

**关键**：① 复用 `_build_sql` 全部现有覆盖作 CTE body（HAVING/窗口/分区top-N/多base/维度全可作 body）② outer arg **alias-based**（引 inner 输出列 = metric name ∪ dimension；严禁 raw caliber/表前缀，同 having R-SL-80）③ count(*) 无 arg / 其他 func 需 arg。

### 2.3 inner LIMIT — outer+lf.limit=0 无 LIMIT（⭐ 守护者 Stage 3 必纠 · 非 cap）

⭐ **守护者 Stage 3 区分 display-cap vs aggregate-input-cap**（纠草案+Stage 2 的「保留 cap + 文档」）：
- **display-cap**（「给我看城市」→ 显示前 1000）：可接受。
- **aggregate-input-cap**（outer 把 inner 喂进 COUNT/SUM）：默认 1000 cap → 「多少城市 GMV>1万」答 1000（实际 1500）= **头号用例的静默错标量答案**（比回退 LLM 更坏）。`cardinality<<1000` 是 kk 数据特定假设（feature 走 OSS；distinct 用户/产品高基维可 >1000）。**no-limit-inner 修复免费、无歧义、正确**（GROUP BY 按维度收敛 inner；outer 仍 1 行；不像 NULL-dim `<=>` 有 Doris 疑问）。

- **outer 存在 + `lf.limit == 0`** → inner **无 LIMIT**（聚合全部 groups，正确）
- **outer 存在 + `lf.limit > 0`**（显式 top-N）→ inner **保留 LIMIT**（top-N 后聚合，如「top-5 合计」intentional）

**实现（非 string-strip — v0.7.9 教训）**：`compile_helpers._order_limit` 加负 limit 哨兵（`lf.limit < 0` → 无 LIMIT 子句）；`compile_logicform` 在 outer+lf.limit=0 时 `replace(lf, limit=-1)` 建 inner。**存量（limit≥0）byte-equal 不受影响**（-1 仅内部 outer-aggregate inner 设；LLM 出 limit≥0）。

### 2.4 parser

`_validate_hit` **不变**（outer 是后处理，命中校验仍 metric ∈ 注册表 + 单/多 base 路由不变）。
`_LOGICFORM_SYS` 铁律 + JSON 格式 +outer：
> - **outer = 结果再聚合**（对编译结果再聚合一层，如「GMV 超 1 万的**城市数**」→ dimensions=[city]+having=[gmv>10000]+outer={func:count}；「top-5 城市 GMV **合计**」→ window+qualify+outer={func:sum,arg:gmv}）：func 白名单 `count|sum|avg|min|max`；count 无 arg；其他 func 的 arg **必须引 metric name 或维度**（alias-based，严禁 raw 口径/表前缀）。无再聚合需求 → 留空 {}。

4. **func 白名单 + arg alias-based**：未知 func/缺 arg/arg 越界 → raise 回退。

6. **inner LIMIT 修复**：outer + limit=0 → inner 无 LIMIT（count groups 不被 cap）；outer + limit>0 → inner 保留 LIMIT（top-N 后聚合）；存量 limit≥0 byte-equal。
