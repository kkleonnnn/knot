## §0 触发与战略意义

v0.6.1.4 HTTP API adapter 上线后发现 3 个稳态债：

| 现象 | 根因（v0.6.1.x 上线追溯）| 守护层级 |
|---|---|---|
| futures_admin.* 表 admin UI 重建后丢 source_type → pick_http_route 静默 fallback SQL | catalog._load_from_db 无 source_type 兜底 | C1 |
| "用户10047725历史持仓" 模糊 entity → 误路由 HTTP 当前持仓表 | pick_http_route 只 keyword 匹配，无 entity 语义判断 | C2 |
| sql_planner LLM 输出"无法直接 SQL 查询 HTTP 虚拟表"中文字符串 → sql_validator fail-open → presenter 当 SQL 展示 | sql_planner 输出非 SELECT 时未拒识 | C3 |

**红线落地源**：v4 LOCKED §3.3（R-PB-C1-1/2 + R-PB-C2-1/2/3 + R-PB-C3-1/2）

## §6 风险与依赖

| 风险 | 缓解 |
|---|---|
| Clarifier 二次确认 prompt 改动可能误判正常问题 | 测试 5 个边界 case（"BTC 持仓" / "用户 X 持仓" / "X 月持仓" / "强平记录" / "实时持仓"）|

| sql_planner 非 SELECT 拒识 false positive | 仅检测起手 token（SELECT/WITH）；不影响 CTE/UNION 等复杂 SQL |
| admin UI 改 source_type 影响既有 doris/mysql 数据源 | 默认 "db"，仅 HTTP 显式选；既有数据源行为 sustained |
