## 1. 决议（首个非编译覆盖方向 · 标量+单层最小可行）

**一句话**：metric 模型从「原子 caliber」扩展到「**二元派生指标**」（A op B，A/B = 已定义原子 metric，op ∈ ÷×+-）—— 解锁占比/人均/率（如 `arpu = gmv ÷ dau`、`cvr = orders ÷ visits`、`毛利率 = profit ÷ revenue`）。

### ⭐ 核心判断（grounded main `3908058`）—— v0.7.0 预留激活

- **metrics.lineage 列 v0.7.0 已为此预留**（prestudy「lineage: 派生指标依赖（如 客单价 = gmv / order_cnt）」；plan「lineage **inert 存储**，无消费者；派生编译 + 循环 DFS 校验**随派生一并留 v0.7.1+**」）→ **本刀正是该激活**：用 lineage 存**结构化派生定义**（opaque JSON 可定结构），**0 schema ALTER**。
- **派生 metric**：`lineage = {"op":"divide", "left":"gmv", "right":"dau"}`（结构化注入安全），`caliber = ""` + `base_object = ""`（无单一 base）。**原子 metric**：lineage = ""（存量 0 漂移）。判别：lineage 非空 → 派生。
- **标量编译**（资深拍范围）：派生 = `(left 标量子查询) <op> (right 标量子查询)`，复用 v0.7.11 标量子查询形态 + 算术；divide → `NULLIF(right, 0)` 除零。
- **单层（资深拍范围）**：left/right 须**原子**（lineage 空）→ 拒 derived-of-derived → **免 DFS**（嵌套 DFS 防循环留后续，v0.7.0 planned）。

```sql
-- arpu = gmv ÷ dau（lineage={op:divide,left:gmv,right:dau}；gmv@orders + dau@users）
SELECT (SELECT SUM(o.pay_amount) FROM shop.orders o WHERE o.status='paid')
     / NULLIF((SELECT COUNT(DISTINCT o.user_id) FROM shop.users o), 0) AS arpu
```

### In-Scope
- metric 模型：lineage 结构化派生定义（激活）+ metric_repo 派生支持/校验
- compiler：派生检测 + `_build_derived_sql`（标量展开 + NULLIF）+ 路由
- parser：`_validate_hit` 派生感知 + prompt 教派生 metric
- AdminMetricRegistry：派生 metric 注册 field（admin UI）

### OOS（后续刀 / 资深范围裁定）
- **维度派生**（各城市 arpu）—— 维度并集驱动（v0.7.12）+ 逐维除法，复杂
- **嵌套派生**（derived-of-derived）—— DFS visited-set 防循环（v0.7.0 planned）
- 多元派生（>2 项）/ 派生 + having/window/qualify/outer 组合

## 2. 设计

### 2.1 metric 模型（lineage 激活 · 0 ALTER）

- **派生 metric**：`lineage = {"op": "<divide|multiply|add|subtract>", "left": "<metric name>", "right": "<metric name>"}`（JSON 存 lineage 列）；`caliber`/`base_object` 空。
- **原子 metric**：lineage = ""（存量不变）。
- `_OP_SQL = {"divide":"/", "multiply":"*", "add":"+", "subtract":"-"}`（op 白名单 → SQL 运算符；注入安全）。

### 2.2 metric_repo 派生支持

- `create_metric` 校验放宽：**caliber 必填 OR lineage 为合法派生**（派生 metric 无 caliber）。
- `_validate_lineage(lineage)`：op ∈ 白名单 + left/right 为非空 str（metric name）；非法 → MetadataError。
- list_metrics 返回 lineage（已有列）；新增 `parse_lineage(m) -> dict|None` helper（解析 JSON）。

### 2.3 compiler 派生展开

`_build_sql` 派生分支（**单 metric 派生路由 + 先于 `_metric_bases`** —— 守护者 Stage 3：L54 空 base raise 误杀派生）：
```python
def _build_sql(lf, metrics_by_name, tables, time_ctx, relations=None):
    if len(lf.metrics) == 1 and _is_derived(metrics_by_name.get(lf.metrics[0], {})):  # 单 metric 派生（混查/多派生 → 落原子路由 graceful 回退）
        return _build_derived_sql(lf, metrics_by_name, tables, time_ctx)
    bases = _metric_bases(...)   # 原子路径（存量 byte-equal）
    ...
```

`_build_derived_sql`（标量 gate · 去 time —— 守护者 Stage 3）：
```python
def _build_derived_sql(lf, metrics_by_name, tables, time_ctx):
    # 禁列去 time（time per-dep 注入已在 _scalar_subquery，无歧义全局窗 → 支持「本月 arpu」）；filters 禁（下推哪 dep 歧义）
    if lf.dimensions or lf.having or lf.window or lf.qualify or lf.outer or lf.filters:
        raise CompileError("派生指标仅标量（维度/having/window/qualify/outer/filters → 回退；维度派生留后续）")
    name = lf.metrics[0]   # 路由保证单 metric 派生
    return _guard("SELECT " + _derived_expr(metrics_by_name[name], metrics_by_name, tables, time_ctx, lf.time) + f" AS {name}")


def _derived_expr(m, metrics_by_name, tables, time_ctx, lf_time) -> str:
    lin = _parse_lineage(m)            # {op, left, right}
    op = _OP_SQL.get(lin["op"])
    if op is None:
        raise CompileError(f"未知派生 op {lin['op']!r} → 回退")
    left_m, right_m = metrics_by_name.get(lin["left"]), metrics_by_name.get(lin["right"])
    if not left_m or not right_m:
        raise CompileError("派生 left/right ∉ 注册表 → 回退")
    if _is_derived(left_m) or _is_derived(right_m):     # ⭐ 单层：left/right 须原子
        raise CompileError("派生 left/right 须原子 metric（嵌套派生留后续）→ 回退")
    # _scalar_subquery 含 time 注入（per-dep date_col，v0.7.11 L46-51 抽出）→「本月 arpu」两 dep 各注入
    left, right = (_scalar_subquery(left_m, tables, time_ctx, lf_time),
                   _scalar_subquery(right_m, tables, time_ctx, lf_time))
    if lin["op"] == "divide":          # 除零防护
        right = f"NULLIF({right}, 0)"
    return f"{left} {op} {right}"
```
`_scalar_subquery(m, tables, time_ctx, lf_time)`（抽 compile_helpers leaf · **带括号无别名** `(SELECT caliber FROM physical o [WHERE filters AND time])` · **含 time per-dep 注入**）：multi_base caller 加 ` AS {name}`（byte-equal endswith 验）；derived caller 包 `{L} {op} NULLIF({R},0)`。

`_scalar_subquery(m, tables)` = `(SELECT <caliber> FROM <physical> o [WHERE <filters>])`（抽 v0.7.11 `_build_scalar_sql` 子查询构造为共享 helper；复用）。

**注入安全**：op 白名单 → SQL 运算符；left/right = metric name → 解析为注册表原子 metric → 其 caliber（admin 定义，同存量信任）；NULLIF 除零；**0 用户原始表达式**。

### 2.4 parser 派生感知

- `_validate_hit`：派生 metric ∈ 注册表即 OK（**派生 metric 跳过 base 校验** —— base_object 空不计入 objs；compiler 再校验 deps 原子）。原子 metric base 校验 sustained。
- `_LOGICFORM_SYS`：注入的 metric 清单含派生 metric（名/display）；LLM 像普通 metric 引用（如「客单价」→ arpu）。prompt 加注派生 metric 可直接引用。

admin 注册屏 +「派生定义」field（op 下拉 + left/right metric 选择）；提交 → lineage 结构化 JSON。VRP 视觉延续。
