## §2 决策点（资深拍板项 · 执行者带推荐）

### D1 — 语义（资深已拍：append-only 恢复）
- 「恢复版本 N」= 用版本 N 的 `logicform_json`（+ `compile_error_reason` 忠实复制源状态）**新建一个修正行**
  （is_corrected=1, parent_message_id=原 message_id），append-only **0 删除/0 修改历史**，链尾=当前采纳。
- AuditAction `logicform.rollback` 留痕。

### D2 — ⭐ 是否加 `restored_from_audit_id` 列（推荐：加）
- **✅ 推荐加列** `restored_from_audit_id INTEGER`（nullable；普通修正=NULL，恢复行=源版本 audit_id）：
  审计血缘自描述（侧表行自带「恢复自 #N」，与 v0.7.3 parent_message_id 同精神）+ 前端版本历史显「恢复自 vN」徽标。
  ALTER ADD COLUMN **幂等**（PRAGMA table_info 检查 / init_db 模式）。
- 备选（不推荐）：不加列，恢复来源仅记 audit_log detail —— 侧表行无法自描述，前端只显「修正」（内容碰巧=源）。

### D3 — 端点（推荐：独立 /restore）
- **✅ `POST /api/admin/logicform-audit/{audit_id}/restore`**（require_admin + 2FA）：源 audit_id 的版本内容新建修正行。
- 与 `/correct`（admin 手改 LF compile-only）、`/rerun`（执行验证）区分：/restore = 用既有版本内容恢复（0 编译 0 执行，纯 append）。

### D4 — 恢复源版本范围（推荐：任意版本 · 忠实复制）
- **✅ 允许恢复任意历史版本**（原始/修正/near-miss）—— append-only 忠实复制源 `logicform_json` + `compile_error_reason`
  （恢复 near-miss → 恢复行也 near-miss，history 端点分层照显；admin 主导，前端可提示「该版本为 near-miss」）。
- 0 重编译（避 v0.7.5 保真度问题）：恢复 = 字节复制源版本状态，非重新判定。

### D5 — audit（mutation 留痕）
- **✅ AuditAction `logicform.rollback` 46→47**；detail 记 `restored_from_audit_id` + `message_id` + `new_audit_id`。
